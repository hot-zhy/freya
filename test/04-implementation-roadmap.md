# MR 风险评估系统实施路线图

## 1. 范围与目标

本路线图用于指导 MR 风险评估系统从 0 到 1，再到规模化优化的落地过程。目标是在不显著拖慢研发效率的前提下，提升变更风险发现能力与测试策略质量。

## 2. 分阶段计划

## Phase 0：准备期（1-2 周）

### 目标

- 明确接入边界与基础数据可用性
- 完成最小设计评审与 Owner 分工

### 关键任务

- 统一 `ChangeEvent`、`RiskAssessment`、`FeedbackRecord` 数据契约
- 盘点历史 200+ 拦截记录字段并完成标准化标签定义
- 对接 MR 平台 API 权限与 webhook 鉴权方案

### 交付物

- 数据字典 v1
- 接口契约草案
- 风险规则清单初稿（Top 20）

### 验收指标

- 数据契约评审通过
- 历史记录可用于检索和统计（结构化率 >= 90%）

## Phase 1：MVP（3-6 周）

### 目标

- 打通从 MR 触发到评估回写的主链路
- 给出可解释风险等级与测试建议

### 关键任务

- WebhookGateway + EventQueue + 分析 Worker 上线
- 实现规则驱动 RiskScoringEngine（不引入复杂模型）
- 实现 TestStrategyComposer（按 L1-L4 模板输出）
- MR 评论和 check 状态回写

### 交付物

- 可运行 MVP 服务
- 规则库 v1（覆盖高频风险）
- 观测面板 v1（时延、错误率、风险分布）

### 验收指标

- 快速结论 P95 <= 60s
- 完整结论 P95 <= 180s
- 解释完整率 >= 95%

## Phase 2：增强期（4-8 周）

### 目标

- 提升准确率与建议可执行性
- 引入历史案例检索与反馈闭环

### 关键任务

- 接入历史拦截记录检索（相似案例引用）
- 增加契约风险、DB 迁移、发布窗口等高级特征
- 建立误报/漏报反馈采集机制
- 对接发布平台，补充灰度观察模板

### 交付物

- KnowledgeRetriever v1
- 反馈看板与规则校准流程
- 策略建议增强版（含灰度与回滚建议）

### 验收指标

- L3/L4 召回率 >= 85%
- 误报率 <= 20%
- 人工标注回流率 >= 70%

## Phase 3：规模化与治理（持续）

### 目标

- 从建议系统升级为可治理的质量门禁能力
- 按业务线差异化策略

### 关键任务

- 引入分业务线阈值校准
- 构建规则版本管理与灰度发布机制
- 对关键系统启用条件式强门禁（L4 必审）
- 增加成本治理（任务并发、缓存命中、重试策略）

### 交付物

- 治理规范（门禁策略、例外流程、审计规范）
- 规则/阈值自动化校准作业
- 质量 ROI 报告（事故下降、回滚下降）

### 验收指标

- 关键系统高风险漏报持续下降
- 系统可用性 >= 99.5%
- 研发满意度显著提升（主观评分或采纳率）

## 3. 里程碑计划（建议）

| 里程碑 | 时间点 | 关键成果 |
|---|---|---|
| M1 | 第 2 周 | 设计冻结、数据契约与规则范围确定 |
| M2 | 第 6 周 | MVP 主链路上线，MR 可见风险评估结果 |
| M3 | 第 10 周 | 历史案例检索 + 反馈闭环上线 |
| M4 | 第 14 周 | 关键业务线试点强门禁 |

## 4. 依赖与协作

- **研发效能团队**：平台接入、流水线编排、回写机制
- **业务研发团队**：规则校验、测试策略可执行性反馈
- **SRE/发布团队**：灰度策略与观测指标标准化
- **质量/测试团队**：用例映射与回归资产维护

## 5. 关键实施策略

- 采用“先建议后门禁”降低组织阻力
- 明确所有风险结论必须附带证据
- 用可视化看板持续证明业务价值（事故率、回滚率）
- 每周小步迭代规则，不做大爆炸发布

## 6. 风险与应对

| 风险 | 影响 | 应对策略 |
|---|---|---|
| 初期误报偏高 | 研发反感，采纳率低 | 软门禁 + 快速标注闭环 |
| 样本质量不足 | 评分不稳定 | 优先治理历史数据标签质量 |
| 外部依赖不稳定 | 评估超时或失败 | 缓存 + 降级 + 重试补偿 |
| 跨团队协同慢 | 进度延误 | 建立固定周会与里程碑验收机制 |

## 7. 成功标准

- 工程视角：稳定、可观测、可扩展
- 业务视角：降低高风险变更事故率
- 组织视角：研发可接受且持续使用
