全量面试题可复述答案（项目绑定版）

目标：每题都能 30 秒答清楚，2 分钟能展开，且能连到你的项目。涉及数字的地方给了可调整区间，按你的真实数据替换即可。

推荐口径：AI Coding 占比可讲 35%~50%（需求拆解/样板代码/测试用例/文档）；核心链路设计、异常兜底、性能瓶颈定位仍由你主导。效率提升可讲 30%~60%，并强调“复杂问题不是替代，而是放大工程师能力”。

一、项目与 AI Coding 实战

Q1：简历上项目 AI coding 占比有多少？

A：我一般讲“AI coding 参与度 35%~50%”。重复性编码和脚手架很适合交给 AI，但架构、边界、可靠性策略、压测和上线兜底是我亲自做。

Q2：你的 AI coding 使用范式/习惯？

A：固定四步：先让它做方案草图，再让它只改小范围，再跑测试和静态检查，最后我做 code review。我的习惯是“让 AI 多提选项，最终决策在人”。

Q3：项目从 0 到 1 和 1 到 100 有什么区别？

A：0-1 重点是验证价值闭环，追求“能跑通”；1-100 重点是稳定性、成本、可观测和协作效率，追求“能长期跑、低成本跑”。

Q4：function call、map、skills，智能体演进过程？

A：演进一般是：纯对话 → function call（工具能力）→ tool map/router（按任务选工具）→ skills（沉淀可复用工作流）→ 多智能体协作。核心变化是从“回答”走向“可执行与可复用”。

Q5：项目中最大难点是什么？

A：我遇到的最大难点是“检索命中率与时延同时优化”。比如在“智慧大师”里，既要多源信息融合提升正确率，又要控制链路延迟。做法是召回分层、重排和异步化，最终在可接受时延内稳定输出。

Q6：项目用了什么大模型？

A：我会按场景说“多模型路由”：主模型负责复杂推理，小模型处理分类与抽取；你可按真实情况替换为 ChatGLM、Qwen、Llama 系。你主页已有 AAAI 项目可明确讲 ChatGLM + LoRA + 多模型投票。

Q7：语言模型在系统里的作用？

A：它是“决策与生成中枢”：把非结构化输入转成可执行计划，结合检索证据输出可解释结果。真正落地时，它必须和检索、规则、工具一起工作。

Q8：怎么评估检索效果？

A：离线看 Recall@k、MRR、NDCG；线上看追问率、纠错率、证据点击率。我的做法是“检索层和生成层分开评”，防止把问题都归咎给模型。

Q9：平时怎么用 AI coding，能提效多少？

A：高频用在单测、样板代码、接口文档、重构建议、日志分析。稳定提效一般在 30%~60%，但我会强调“越复杂越需要人工做约束与验收”。

二、Agent / LLM 深挖（含你给的 13 个进阶问题）

Q10：Agent 的 System Prompt 怎么设计与迭代？是否做过自动优化？如何补全不完整意图？

A：我把 System Prompt 分为角色、边界、工具协议、失败回退四层，版本化管理。迭代靠“失败样本回放 + A/B”。自动优化可用提示词模板搜索或规则生成。用户意图不完整时先做槽位提取，再反问关键缺失参数，不盲猜高风险动作。

Q11：构建 Agent 的瓶颈？LangChain memory 在多用户并发有什么问题？

A：瓶颈通常在上下文膨胀、工具不稳定和状态错乱。LangChain 默认 memory 若全局实例化，容易串会话；并且 token window 处理不好会超长崩溃。我的做法是按 userId/sessionId 隔离会话存储，摘要压缩历史，并加 TTL。

Q12：并发用户如何隔离？怎么保证线程安全？

A：核心是“状态不共享、共享必加锁”。会话状态放 Redis/DB，以 sessionId 隔离；进程内只留无状态组件。并发更新用 CAS 或分布式锁，避免同一会话多路写冲突。

Q13：微调 Llama2 如何选样本和清洗？哪些数据质量问题影响大？

A：我会优先选“真实任务闭环数据”，覆盖成功样本与失败修复样本。清洗包括去重、去脏、脱敏、统一指令格式。影响最大的脏数据是：错误答案被当正样本、标签风格不一致、模板泄漏，这会直接导致模型输出漂移和幻觉增多。

Q14：DPO 相比 SFT 的优劣？Agent 场景提升明显吗？偏好对怎么构造？

A：SFT 稳、简单、便宜；DPO 能更直接对齐偏好，减少“看起来对但不可用”的回答。Agent 场景里，DPO 对工具选择和拒答边界常有增益，但依赖高质量偏好对。偏好对可以规则自动生成 + 人工抽检校准，纯自动会引入系统偏差。

Q15：为何选 vLLM？KV-cache 如何加速？你做过哪些优化？

A：选 vLLM 主要看吞吐和调度能力，尤其 PagedAttention 对 KV 内存利用率好。KV-cache 复用历史 token 的 K/V，避免重复计算。常见优化是动态批处理、长度分桶、缓存热门前缀、按请求价值做模型路由。

Q16：要支持 Streaming，但当前延迟超标，如何折中？

A：我会把“首 token 时延”和“全量完成时延”分开优化。先用小模型快速给草答并流式返回，再异步补充高质量段落；同时控制上下文长度和检索候选数，必要时对低优先级请求降级。

Q17：多轮上下文状态管理怎么做？高并发下一致性如何保证？

A：短记忆放会话窗口，长记忆放结构化存储。更新策略采用事件序号或版本号，保证乱序写入可检测。关键写路径走单写者策略或乐观锁重试，避免覆盖。

Q18：你们 Agent 调了多少外部工具？如何做容错与超时？

A：面试可讲 6~12 个工具是常见规模。每个工具设超时、重试、熔断和降级策略；链路层有 traceId。对于高风险工具加人工确认和只读模式，保证失败可控。

Q19：工具调用失败后的 feedback 策略？

A：我会让 Agent 先分类失败类型：参数错误、权限错误、下游超时、数据缺失。然后分别走“自动修参重试 / 切换备选工具 / 友好降级答复 / 请求用户补充信息”，避免一句“失败了”结束。

Q20：从用户行为日志抽取训练对话，如何归一化和事件抽象？

A：先把日志转成统一事件流（提问、检索、工具调用、反馈、改写）。再做脱敏、去重、会话切分、意图标签化，最终形成监督样本或偏好对。关键是保留失败上下文，这比纯成功样本更有训练价值。

Q21：时间窗口/偏移限制对话系统，模型如何“理解时间”？

A：模型本身不可靠地“感知当前时间”，要靠系统注入时间锚点和可检索时间线。工程上会把时间约束写进工具参数和检索过滤条件，而不是只靠自然语言描述。

Q22：你觉得 Agent 哪些模块最容易在生产出问题？如何监控定位？

A：最容易出问题的是工具编排、上下文污染和权限边界。监控上我会盯四类指标：成功率、端到端延迟、工具超时率、人工接管率；定位靠全链路日志和失败回放。

Q23：什么是 Agent / MCP / Skill？

A：Agent 是可规划可执行的智能体；MCP 是模型与外部工具/资源的标准协议；Skill 是可复用任务模板（步骤、约束、参数），让 Agent 稳定复用最佳实践。

Q24：Transformer 是什么？

A：核心是自注意力，能并行建模 token 之间关系，配合位置编码与前馈网络实现强表达能力。它成为主流因为可扩展、可并行、在大数据大算力下效果稳定上升。

Q25：从安全角度看，用户↔Agent、Agent↔Agent、Agent↔后端有哪些风险？

  • 用户↔Agent：Prompt 注入、越权指令、敏感信息诱导泄漏。
  • Agent↔Agent：上下文污染、信任链被劫持、任务边界漂移。
  • Agent↔后端：工具越权调用、参数注入、横向访问未授权资源。

防线：最小权限、参数白名单、敏感操作二次确认、审计日志、隔离执行沙箱。

Q26：MCP 和 Shell 脚本实现功能有什么区别？

A：Shell 脚本偏“过程自动化”，灵活但难标准化治理；MCP 偏“能力标准化暴露”，有清晰工具 schema、权限边界和可观测。团队化和多模型复用场景，MCP 维护成本更低。

Q27：AI 时代对传统后端有什么影响？是不是不用懂后端了？

A：影响是后端职责从“纯接口实现”升级为“AI 系统编排与稳定性负责人”。更需要懂并发、事务、一致性、缓存、观测、成本。不会后端，只会调模型，很难把系统做稳。

三、Java / Spring / 并发 / 数据库高频

Q28：AOP 原理？

A：通过动态代理在目标方法前后织入增强逻辑。Spring 会在 Bean 初始化后生成代理对象，统一处理事务、日志、权限等横切关注点。

Q29：JDK 动态代理和 CGLIB 区别？

A：JDK 基于接口，轻量；CGLIB 基于继承，能代理无接口类，但不能代理 final 类/方法。Spring 默认有接口用 JDK，无接口用 CGLIB。

Q30：事务传播行为？

A：重点记 REQUIRED（默认，加入当前事务）、REQUIRES_NEW（新开事务，挂起外层）、NESTED（保存点回滚）。面试常追问“为什么子事务异常不影响外层”，要结合传播行为解释。

Q31：BeanFactory 和 FactoryBean 区别？

A：BeanFactory 是 IoC 容器；FactoryBean 是“产 Bean 的工厂 Bean”。从容器拿 myFactoryBean 默认拿到其生产对象，拿工厂本身要用 &myFactoryBean。

Q32：介绍分布式事务？

A：跨服务场景通常不追求强一致锁死全局，而是走最终一致。常见方案是本地消息表、事务消息、SAGA/TCC。实践重点是幂等、补偿、可观测。

Q33：什么情况下出现事务悬挂？

A：常见于 TCC，空回滚先执行，后续 Try 到达但上下文已失效，导致业务状态异常。处理方式是幂等表 + 防悬挂标记 + 状态机校验。

Q34：介绍分布式锁？

A：Redis 常见实现是 SET key value NX EX，释放时用 Lua 保证“比对 value + 删除”原子。生产要有续期机制、超时与失败降级，避免死锁和误删。

Q35：Redis 过期删除策略？

A：惰性删除 + 定期删除。惰性在访问时删，省 CPU；定期抽样扫描，防止过期 key 堆积。内存淘汰策略再兜底。

Q36：缓存雪崩、缓存风暴（击穿）、缓存穿透？

A：雪崩是大量 key 同时失效；击穿是热点 key 失效瞬间并发打 DB；穿透是查不存在数据。治理：随机 TTL、单飞锁、布隆过滤器、空对象缓存、限流熔断。

Q37：MQ 生产者提交超时怎么办？

A：先区分“没发出去”还是“已落库但回执丢失”。业务层必须做幂等重试，并记录消息状态表。必要时开启事务消息或回查机制，防止重复下游副作用。

Q38：怎么保证消息幂等？

A：消费端以业务唯一键去重（如订单号+事件类型），插入幂等表或利用唯一索引。处理流程要做到“先判重再执行业务再落处理标记”。

Q39：synchronized 底层原理？

A：底层依赖对象头 Mark Word 和 Monitor，字节码是 monitorenter/monitorexit。锁会经历偏向锁、轻量级锁到重量级锁升级，竞争激烈时线程阻塞挂起。

Q40：synchronized 锁静态方法和非静态方法区别？

A：非静态锁的是实例对象；静态方法锁的是 Class 对象。不同实例的非静态锁互不影响，但静态锁是类级别全局互斥。

Q41：ReentrantLock 原理？

A：基于 AQS，核心是 state + CLH 队列。抢锁失败入队并挂起，释放时唤醒后继；支持可中断、公平策略和多个条件队列。

Q42：怎么实现可重入？

A：同线程重复获取锁时仅增加重入计数（AQS state++），释放时递减，减到 0 才真正释放锁。

Q43：公平锁和非公平锁怎么实现？

A：公平锁先检查等待队列前驱，严格 FIFO；非公平锁允许新线程先 CAS 抢锁，吞吐更高但可能饿死部分线程。

Q44：线程池参数与工作原理？

A：核心参数是 corePoolSize/maxPoolSize/queue/keepAliveTime/rejectHandler。执行流程：核心线程未满先建线程，满了先入队，队列满再建非核心，仍满触发拒绝。

Q45：拒绝策略有哪些？

A：Abort、CallerRuns、Discard、DiscardOldest。生产里常自定义：记录告警、按任务等级降级、关键任务转 MQ。

Q46：核心线程能回收吗？

A：默认不回收；开启 allowCoreThreadTimeOut(true) 后可按 keepAlive 回收，适合流量波动大且要控成本的场景。

Q47：JUC 原子类如何实现原子性？

A：依赖 CPU CAS 指令 + volatile 内存可见性。失败就自旋重试，冲突高时可用 LongAdder 降低热点竞争。

Q48：CAS 会出现什么问题？

A：ABA、长时间自旋消耗 CPU、只能保证单变量原子性。ABA 常用版本号（AtomicStampedReference）解决。

Q49：MySQL 悲观锁与乐观锁？

A：悲观锁如 SELECT ... FOR UPDATE，适合高冲突；乐观锁靠版本号，适合读多写少。高并发写热点通常悲观锁更稳。

Q50：索引失效场景？

A：对索引列做函数/计算、隐式类型转换、最左前缀破坏、like '%xx'、范围查询后列失效、数据分布导致优化器放弃索引。

Q51：INT 索引什么时候会失效？

A：最常见是和字符串比较导致隐式转换，如 where int_col='123a'，可能触发全表扫描；还有函数包裹 int_col+1=10 也会失效。

Q52：索引有哪些数据结构？

A：常见有 B+Tree（MySQL InnoDB 主流）、哈希（等值快，范围弱）、倒排索引（全文检索）、跳表（Redis 有序集合）。

Q53：B+ 树结构是怎样的？

A：非叶子节点只存键和指针，叶子节点存完整数据并链表相连。好处是层高低、磁盘 IO 少、范围查询和排序非常高效。

Q54：平时怎么学技术？

A：我用“三段法”：先用项目问题驱动找资料，再做最小可运行实验，最后复盘沉淀成题库和笔记。这样知识能从“看懂”变成“讲清+用得出”。

四、可直接讲的项目绑定话术（建议背）

话术 1：AI coding 占比与价值

“我在项目里 AI coding 占比大约 40% 左右，主要放在样板代码、测试与文档。真正影响上线稳定性的架构决策、异常兜底、压测调优是我亲自做，所以它对我是效率放大器，不是替代工程能力。”

话术 2：智慧大师项目难点

“最大难点是多源信息融合后，检索准确率和时延很难同时优化。我把链路拆成召回、重排、生成三段，召回用 Milvus 提覆盖，重排提精度，再用 RabbitMQ 做异步化，最后把高并发抖动压住了。”

话术 3：后端在 AI 时代的定位

“AI 时代后端更重要了，因为模型只是能力源，真正把能力变成稳定产品靠后端做编排、缓存、事务、一致性和可观测。不会后端，很难把 Agent 系统做成生产级。”

六、深挖增强版（更可靠、更抗追问）

增强 1：AI Coding 占比怎么说才真实可靠？

30秒版本：“我把 AI coding 控制在 35%~50%，主要承担脚手架、测试和文档，关键设计和上线责任仍由我承担。”

2分钟深挖：我把研发任务拆成“可自动化”和“高风险决策”两类：前者交给 AI，后者自己做。高风险包括事务边界、并发控制、压测瓶颈、告警阈值、灰度与回滚。这样做能确保提效的同时不牺牲稳定性。面试官追问时，最好补一句“我会记录 AI 生成代码占比和返工率”。

项目绑定：你可结合“小花狮”说：接口骨架和测试样例由 AI 协助，Redisson 锁策略、MQ 重试和幂等链路由你拍板。

增强 2：0-1 与 1-100 的工程差异（可画架构图）

30秒版本：0-1 看价值闭环，1-100 看稳定性与成本效率，关注点从“功能可用”转为“SLA 可承诺”。

2分钟深挖：0-1 阶段重点是需求真假验证、最小可用链路、快速迭代；1-100 重点是容量评估、扩缩容策略、可观测体系、故障演练和跨团队协作。技术动作包括：限流熔断、降级、缓存层次化、异步化削峰、读写隔离、灰度发布。组织动作包括：On-call、事故复盘、变更审批和回滚预案。

项目绑定：“智慧大师”可讲 0-1 先验证检索可用，1-100 再做多源融合 + 异步化 + 指标监控。

增强 3：Agent 安全链路（用户-Agent-工具-后端）

30秒版本：安全核心是最小权限、输入输出校验和审计留痕，绝不让模型直接拥有无限执行权限。

2分钟深挖：用户到 Agent 的风险是 prompt 注入和越权诱导；Agent 到 Agent 的风险是上下文污染与信任链被劫持；Agent 到后端是参数注入和横向越权。治理分四层：1）鉴权与作用域令牌；2）工具 schema 白名单校验；3）高风险动作强制人工确认；4）全链路 traceId 和审计。补充一句“我会做只读模式和模拟执行模式（dry-run）”，可信度更高。

项目绑定：可说你把高风险工具单独隔离，并限制写权限只对特定 API 开放。

增强 4：MCP 与 Shell 脚本不是替代关系，而是分层关系

30秒版本：Shell 擅长一次性自动化，MCP 擅长标准化能力暴露和治理，适合多模型、多团队复用。

2分钟深挖：Shell 的问题是输入输出协议不统一、权限边界难管理、可观测性弱。MCP 的优势是工具有 schema、参数类型、权限边界和资源接口规范，方便接入审计、限流和策略控制。生产里常见做法是“底层还是脚本/服务，上层通过 MCP 统一封装”。这样既保留灵活性，也提升治理能力。

项目绑定：可讲你在多源工具接入时优先走标准化协议，减少 ad-hoc 脚本维护成本。

增强 5：vLLM、KV-cache、Streaming 延迟取舍怎么答得专业

30秒版本：vLLM 主要优势是吞吐和显存利用率；KV-cache 减少重复计算；Streaming 优化的是感知时延，不等于总时延降低。

2分钟深挖：我会把指标拆成 TTFT（首 token 时延）、TPOT（每 token 时间）和 E2E 时延。KV-cache 主要降低 TPOT，动态批处理提升吞吐但可能拉高 TTFT，所以要按业务类型分池路由。Streaming 超标时的折中是：短答先流式返回，复杂补充异步生成；同时限制上下文长度、减少低价值检索候选、按优先级做降级。

项目绑定：你可以讲“课堂助手”里优先保证首屏响应，再延迟补全解释细节。

增强 6：DPO vs SFT 如何回答“不是背概念”

30秒版本：SFT 解决“会不会做”，DPO 解决“更偏好怎么做”。两者通常串联，不是二选一。

2分钟深挖：SFT 成本低、训练稳定，适合先把任务能力拉起来；DPO 用偏好对做对齐，能改善风格、拒答边界和工具调用选择。偏好对构造建议“规则自动初筛 + 人工抽样校准 + 线上反馈闭环”，否则偏好噪声会被放大。面试时提一句“我看重偏好数据的一致性而不是数量”，更像真实实践者。

项目绑定：可用 AAAI 项目延展：先 SFT/LoRA 提能力，再通过偏好策略优化答案可读性和步骤完整性。

增强 7：事务、MQ、幂等三件套（后端追问高频）

30秒版本：我默认 MQ 是至少一次投递，所以业务端必须幂等；跨服务一致性优先用最终一致，而不是重锁全局事务。

2分钟深挖：生产者超时时要区分“发送失败”和“发送成功但回执丢失”。消费端采用业务唯一键 + 幂等表或唯一索引；状态变更语句加条件，例如 WHERE status='INIT'。跨服务推荐本地消息表/事务消息/SAGA，配补偿和对账任务兜底。面试官追问事故时，可说“我们监控重复消费率和补偿成功率”。

项目绑定：“小花狮”异步批改链路可直接套这套话术。

增强 8：并发锁与线程池怎么讲到“可落地”

30秒版本：锁的选择看冲突密度和可中断需求，线程池调优看队列、拒绝策略和任务类型，不看单一参数。

2分钟深挖：synchronized 适合简单互斥；ReentrantLock 适合可中断、可超时、条件变量场景。线程池调优先看任务性质（CPU/I/O 密集），再定队列与最大线程，拒绝策略必须业务化：关键任务转 MQ，非关键任务降级。再补“我们会监控队列长度、任务等待时长、拒绝率和线程活跃度”，可信度很高。

项目绑定：可讲你在高并发接口通过异步化和限流保护核心线程池，避免雪崩。

增强 9：缓存三大问题（穿透/击穿/雪崩）如何答成体系

30秒版本：穿透防非法请求，击穿防热点失效瞬时流量，雪崩防大量 key 同时失效，三者治理策略不同。

2分钟深挖：穿透用参数校验 + 布隆过滤器 + 空值缓存；击穿用 singleflight 或分布式互斥重建；雪崩用随机 TTL、多级缓存、限流熔断和 Redis 高可用。面试加分点是“缓存策略必须配套降级策略”，例如返回兜底内容或近似值，避免直接打爆 DB。

项目绑定：可结合“智慧大师”说热点知识条目用永不过期 + 异步刷新。

增强 10：索引失效与 B+Tree，怎么说出工程判断力

30秒版本：B+Tree 适合磁盘和范围查询，索引失效常由写法问题和优化器选择导致，先看执行计划再改 SQL。

2分钟深挖：索引失效高频原因：函数包裹、隐式类型转换、最左前缀破坏、范围查询后续列失效、like '%xx'。排查流程：先 EXPLAIN 看走没走索引，再看基数和回表成本，再决定是改 SQL、补联合索引还是强制 hint（慎用）。面试时强调“索引优化是查询模式驱动，不是索引越多越好”。

项目绑定：可讲你在高频查询上通过联合索引和 SQL 改写把慢查询降下来。

面试实战建议：每题按“结论一句话 → 机制两点 → 项目证据一条 → 风险兜底一条”回答，基本就能抗住大多数追问。

七、RAG 与 Agentic RAG 深入专题（已加可复述版）

RAG 全流程优化：总览

一条线讲清：数据接入 → 分块 → 向量化与索引 → 查询理解 → 召回重排 → 上下文构建 → 生成约束 → 反馈闭环。

答题模板：“我做 RAG 不是只调 embedding，而是按链路分层优化；召回层看覆盖，生成层看可证据化，线上看时延和成功率。”

RAG 流程 1：数据接入与清洗（Ingestion）

  • 目标：把原始文档变成可检索、可追踪、可更新的知识资产。
  • 优化：去页眉页脚/模板噪声、保留标题层级和来源字段、做版本号和增量更新。
  • 坑点：只存纯文本不存 metadata，后续权限过滤和时间过滤会很难做。

RAG 流程 2：分块策略（Chunking）

  • 目标：让每个 chunk 既有语义完整性，又能被精准召回。
  • 优化：优先按语义边界切（标题、段落、函数），再控 token 大小并设置 10%~20% overlap。
  • 经验：FAQ、代码、规范文档应使用不同切分策略，避免“一刀切”。

RAG 流程 3：Embedding 与索引构建

  • 目标：兼顾召回率、时延和成本。
  • 优化：同域 embedding、ANN 参数调优（如 HNSW/IVF）、冷热分层索引。
  • 坑点：embedding 模型切换后不重建索引，会导致历史向量语义漂移。

RAG 流程 4：查询理解与改写（Query Understanding）

  • 目标：把用户自然语言转成“可检索表达”。
  • 优化：query rewrite、实体补全、同义扩展、多路并行查询。
  • 追问可答：复杂问句可先拆子问题再检索，提升多跳问题命中率。

RAG 流程 5：召回 + 重排（Retrieve + Rerank）

  • 目标：先高覆盖，再高精度。
  • 优化：混合检索（BM25 + Vector）并行召回，cross-encoder 重排到 topK。
  • 工程点：先 metadata 过滤再重排，可减少重排开销并提升稳定性。

RAG 流程 6：上下文构建（Context Assembly）

  • 目标：给模型最少噪声的证据包。
  • 优化：证据去重、相邻片段合并、按可信度排序、严格 token budget。
  • 加分点：每条证据附 source id，支持可追溯引用。

RAG 流程 7：生成约束与拒答策略

  • 目标：答案正确、可解释、可拒答。
  • 优化：提示词强约束“仅基于证据回答”，证据不足时触发拒答或补检索。
  • 坑点：没有拒答策略会导致“看起来流畅但事实上幻觉”。

RAG 流程 8：反馈闭环与持续优化

  • 目标：系统越跑越准，而不是上线后停滞。
  • 优化：沉淀失败样本（没召回/召回错/生成错），做离线回放与定期回归评测。
  • 实践：优先优化检索和重排，不要一上来就改大模型。

Agentic RAG 范式 1：ReAct RAG

做法：Thought → Retrieve → Observe → Refine → Answer，允许多轮检索迭代纠错。

适用：多跳推理、跨文档证据融合。控制点：设置最大轮次和超时，防止无限循环。

Agentic RAG 范式 2：Plan-and-Execute RAG

做法：先拆子任务，再对子任务独立检索并汇总。

适用：长问题、报告生成、方案比较。控制点：子任务输出统一 schema，便于最后合并。

Agentic RAG 范式 3：Router RAG（路由式）

做法：先分类 query，再路由到不同知识源（FAQ、代码库、工单库、政策库）。

适用：多知识域组织。控制点：路由失败要 fallback 到通用检索。

Agentic RAG 范式 4：Self-RAG / CRAG（自反思）

做法：模型先回答，再自评证据充分性；不足则触发补检索和重生成。

适用：准确性优先场景。控制点：自评阈值要标定，否则检索次数失控、时延上升。

Agentic RAG 范式 5：Multi-Agent RAG

做法：Planner 拆解，Retriever 取证，Verifier 校验，Writer 汇总。

适用：复杂任务和强可解释输出。控制点：角色边界、共享状态一致性、审计日志必须明确。

Agentic RAG 范式 6：GraphRAG（图谱增强）

做法：图结构检索关系链 + 文本检索事实证据，最后联合推理。

适用：关系密集型问题（依赖链、组织关系、因果链）。代价：图构建与更新成本较高。

RAG 评估体系：检索层 / 生成层 / 端到端分开评

  • 检索层：Recall@k、MRR、NDCG、HitRate、P95 时延。
  • 生成层：Correctness、Faithfulness（是否被证据支撑）、Citation Accuracy、Completeness。
  • 端到端：任务成功率、追问率、人工接管率、单问成本、用户满意度。

关键原则：先定位“没召回”还是“召回对但生成错”，再做针对性优化。

RAG 评估怎么落地：离线 + 线上 + 回放

  • 离线：构建基准集（问题、金标准证据、标准答案），做回归测试。
  • 线上：A/B 对比不同检索链路，观察真实业务指标变化。
  • 回放：把线上失败会话重放到新策略，验证是否实质提升。

补充：LLM-as-Judge 可用，但要配人工抽检，避免评估偏差。

可直接背诵：RAG 三档口述（30秒 / 2分钟 / 5分钟）

30秒：“我做 RAG 会按数据、检索、生成三层优化：检索层先看 Recall@k 和 MRR，生成层看 groundedness 和引用正确率，线上看时延与成功率。复杂问答会用 Agentic RAG 迭代检索纠错。”

2分钟：“我把链路拆成数据清洗、chunk、混合召回、重排、上下文构建、生成约束、反馈回放。先保召回覆盖，再提重排精度，最后控制 token budget 和拒答策略。评估分层做，避免把问题都归因给模型。”

5分钟：“我会补充范式取舍：ReAct 适合多跳，Plan-Execute 适合长任务，Router 适合多知识域，Self-RAG 适合准确性优先，Multi-Agent 适合复杂协作。并说明如何权衡时延、成本、准确率和可观测性。”
