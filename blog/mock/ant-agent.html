一、简历项目深挖

Q1：NVIDIA 项目中，Batch RCA Pipeline 的 AI SubAgent 调度机制是怎样的？滑动窗口并发是怎么设计的？进阶

A：面向 1000+ 高频 failure pattern，按频次排序后以滑动窗口（如窗口大小 N=5）并发调度 SubAgent。每个 SubAgent 独立处理一个 pattern：先做上下文预处理（日志截取 + 版本归因），再调用 LLM 追踪 failure sequence、定位关联源码函数，最终输出结构化 root cause report。窗口满时等待任意一个完成后滑入下一个。好处：控制 LLM 并发数避免限流、保证资源利用率，同时每个 SubAgent 拥有独立上下文不会相互干扰。

Q2：Batch RCA 的幂等恢复与断点续跑是怎么实现的？如果一个 SubAgent 中途失败怎么办？深追问

A：每个 pattern 的分析状态持久化到 SQLite/JSON：pending → running → completed/failed。重启后扫描状态表，跳过 completed，从 failed/pending 恢复。幂等性靠 pattern_id 唯一键保证——同一 pattern 多次执行只覆盖写入。SubAgent 失败时记录错误日志并标记 failed，不阻塞窗口中其他任务，最终生成汇总报告时标注未完成项。

Q3：喜马拉雅项目中，Mem0 长期记忆的写入策略是什么？什么信息值得写入，什么不写？进阶

A：写入门槛设计：①用 LLM 对每轮对话做结构化抽取，判定是否包含用户偏好变更、关键事件（如情绪波动、重要决策）、新兴趣主题；②对抽取结果做去重和冲突检测——若与已有记忆矛盾则更新而非追加；③日常闲聊、纯问候等低信息密度内容不写入。Mem0 同时维护向量索引和结构化字段（时间戳、标签），支持按语义和条件检索。读取时按相关度 + 时间衰减加权，避免过时记忆污染当前决策。

Q4：MCP 在喜马拉雅项目中具体封装了哪些资源？和直接 Function Calling 相比优势在哪？进阶

A：MCP 封装了四类资源：账号信息（年龄/VIP 等级）、行为特征（收听时长/偏好类目）、Mem0 长期记忆（历史对话摘要/关键事件）、外部工具（内容推荐 API/情绪分析服务）。优势：①标准化协议统一工具描述格式，Agent 不用为每个工具写不同的 prompt；②资源可发现——新增数据源只需注册到 MCP Server，Agent 自动获取工具列表；③权限隔离——每个 MCP Server 可独立鉴权，比在 System Prompt 中硬编码 Function Calling 更安全、更易维护。

Q5：喜马拉雅的 RabbitMQ 异步化具体削峰了哪些步骤？消息丢失怎么保证？深追问

A：异步化的高延迟步骤：①用户画像特征抽取（LLM 调用）②语义向量化写入 Milvus ③Mem0 记忆同步。消息可靠性保障：生产端开启 publisher confirm；消费端手动 ACK，处理成功后才确认；配置死信队列（DLX）兜底失败消息；关键消息持久化到磁盘。高峰期 Agent 工具调用超时率降低约 60%，因为这些耗时操作从同步链路中剥离，Agent 只需发消息后立即返回。

Q6：Milvus 中向量召回是怎么做的？聚类 + 情绪标签如何实现兴趣分群？进阶

A：用户画像和 Mem0 关键记忆统一用 Embedding 模型向量化后存入 Milvus。召回时以当前对话向量做 ANN 检索（HNSW 索引），返回 top-K 相关记忆。兴趣分群：离线对全量用户向量做 K-Means / DBSCAN 聚类，每个簇标注主题标签（如"职场焦虑""育儿""历史"）；同时结合情绪标签（正面/负面/中性）做二级划分。在线推荐时，根据用户所属簇 + 当前情绪 + 实时偏好做多维融合召回，点击率提升 8%-12%。

Q7：小花狮的 Reflection 闭环——"大模型生成建议 → 小模型评分校验 → 修订优化"——具体怎么做的？小模型评什么？进阶

A：大模型（如 GPT-4）生成作文批改建议（评语 + 评分 + 修改建议）。小模型（轻量分类/打分模型）对生成结果做质量校验：①格式合规性（是否包含必需字段）②评分一致性（与历史同类作文分数偏差是否过大）③内容安全（是否含不当内容）。若校验不通过，将小模型反馈注入 prompt 重新生成（最多 2 轮）。Redisson 分布式锁保证同一篇作文不会被并发重复批改。这种大小模型配合的 Reflection 模式兼顾生成质量和成本。

Q8：小花狮的高频接口性能治理做了哪些？P99 降到 100ms 以内的关键手段？基础

A：四个关键手段：①Caffeine 本地缓存 + Redis 分布式缓存构成多级缓存，热点数据（班级列表、作业列表）优先命中本地，穿透时查 Redis，再穿透才查 DB；②慢 SQL 排查——通过 EXPLAIN 分析，对高频查询加复合索引、改写子查询为 JOIN；③Lua 脚本实现令牌桶限流，防止突发流量打穿缓存；④RabbitMQ 异步解耦批改等重操作，主接口只做轻量查询。核心思路：读多写少场景以缓存为主，写操作异步化。

Q9：科研助手中，长文本超出 LLM 上下文窗口时的"分块 → 并行 → 聚合"策略怎么设计的？如何处理跨段落的知识丢失？进阶

A：按章节/段落切分论文，每段附上标题和全局摘要作为上下文锚点，并行调用 LLM 生成局部结构化 JSON（节点 + 边 + 层级）。二次聚合阶段：①基于节点名称做去重合并（编辑距离 < 阈值视为同一概念）②解决跨段冲突（如同一术语在不同段被归入不同层级，以频次或后出现为准）③补充段间关联边（如段 A 提到的方法被段 B 引用）。全局摘要注入每段的 prompt 是关键——它让每个 LLM 调用都能感知全文主旨，减少碎片化。

Q10：Redis Stream + SSE 断点续推的消费位点记录是怎么做的？多实例并发消费如何保证不重复不丢失？深追问

A：每个任务/会话维度维护一个 Consumer Group，消费位点由 Redis Stream 的 XREADGROUP 自动管理（Pending Entries List）。客户端断线重连时用 XREADGROUP ... ID 0 先读取 PEL 中未 ACK 的消息，处理后 XACK 确认，再读取新消息。多实例并发：Consumer Group 天然支持竞争消费（同一消息只会被一个消费者拿到）；对于需要严格有序的场景，用 SET NX EX（Redisson 分布式锁）保证同一任务只有一个实例在处理，锁过期后其他实例可接管。

Q11：AI 面试系统的 RAG 链路中，Query Rewrite + 短 Token 命中判定是怎么配合的？为什么不直接用语义检索？进阶

A：纯语义检索对短查询（如"Spring AOP"）的召回效果不稳定——向量空间中短文本信息量不足。做法：①先用 LLM 将短/模糊查询改写为完整的检索语句（如"Spring AOP 的原理、使用场景和与 AspectJ 的区别"）②改写后做向量检索 top-K③对候选结果做短 Token 物理命中判定——原始查询中的关键词是否在候选文档中出现，以此过滤语义相似但主题不匹配的文档。两阶段互补：语义检索保证召回率，关键词命中保证精确度，准确率提升 30%+。

Q12：AAAI 数学竞赛中，K-Means 聚类 + Few-shot 选择策略的具体流程？为什么不直接随机选例子？基础

A：将数学题向量化后做 K-Means 聚类（如 8-10 个簇，对应代数、几何、概率等题型）。推理时先判定当前题目属于哪个簇，从该簇中选择 2-3 个高质量 Few-shot 样例注入 prompt。随机选择的问题：不同题型的解题思路差异大，随机样例可能引入噪声——比如给几何题配一个概率题的解题过程，反而误导模型。题型感知的 Few-shot 让模型看到"同类题该怎么解"，推理质量显著提升。

二、Agent 架构与设计

Q13：Agent 和 Workflow 的本质区别是什么？什么时候该用 Agent，什么时候用 Workflow？基础

A：核心区别在于决策权归属。Workflow：所有分支/步骤由开发者预定义，执行路径确定，适合流程固定的业务（审批流、数据 ETL）。Agent：决策权在模型运行时，模型根据环境反馈动态规划下一步，适合开放性任务（用户意图不确定、工具调用组合不固定）。实际中常混合使用——外层用 Workflow 编排大阶段（如"理解 → 检索 → 生成 → 校验"），每个阶段内部用 Agent 做灵活决策。

Q14：ReAct、Plan-and-Solve、Reflection 三种 Agent 范式各适用什么场景？基础

A：ReAct（Reasoning + Acting）：每步都交替推理和行动，适合工具调用密集、需要即时反馈的场景（如搜索→分析→再搜索）。Plan-and-Solve：先全局规划再逐步执行，适合复杂多步任务（步骤间有依赖、需要控制 token 成本），失败可定位到具体步骤。Reflection：生成 → 自我评估 → 修订，适合对输出质量要求高的场景（写作、代码生成），通过多轮反思提升结果可靠性。

Q15：多 Agent 协作有哪些常见模式？如何防止 Agent 之间的无穷对话？进阶

A：常见模式：①Supervisor（协调者 + 专家）：一个主 Agent 拆解任务分发给专业 SubAgent，汇总结果；②竞争式：多个 Agent 独立解题，投票/融合选最优；③流水线：Agent 串行传递，每个负责一个阶段；④辩论式：Agent 互相质疑和修正。防止无穷对话：设置最大轮次（如 5 轮）、token 预算上限、收敛检测（连续两轮输出相似度 > 阈值则终止）、超时熔断。

Q16：Agent 工具调用失败怎么处理？设计一个健壮的容错机制。进阶

A：分层容错：①重试层：指数退避重试（最多 3 次），处理瞬时故障（网络抖动、限流）；②降级层：主工具不可用时切换备选工具（如主搜索引擎挂了用备选）或返回缓存结果；③反馈层：将失败信息（错误码 + 描述）注入 Agent 上下文，让 Agent 自主决定换策略（如改写查询、换工具、跳过该步骤）；④熔断层：某工具连续失败超阈值后短路，直接走降级逻辑避免资源浪费。所有调用记录审计日志，便于事后分析。

Q17：Agent 的记忆系统应该怎么分层设计？短期 vs 长期 vs 工作记忆？进阶

A：三层记忆：工作记忆——当前任务的 scratchpad（中间推理步骤、工具返回结果），随任务结束清除；短期记忆——当前会话的对话历史，用滑动窗口或摘要压缩管理上下文长度；长期记忆——跨会话持久化的用户画像、偏好、历史事件，存储在向量数据库 + 结构化存储中。写入长期记忆需要门槛（信息密度判断、去重、冲突检测），读取需要相关性过滤 + 时间衰减。关键是避免噪声记忆污染决策。

Q18：如何设计 Agent 的可观测性体系？出了问题怎么定位是哪个环节？深追问

A：四个维度：①分布式追踪——每次 Agent 执行生成唯一 trace_id，每步（推理/工具调用/记忆读写）记录 span，形成调用链；②结构化日志——记录每步的输入 prompt（截断）、输出、耗时、token 消耗、工具参数和返回值；③指标监控——成功率、平均步数、总耗时、token 成本、工具调用分布；④回放与调试——支持按 trace_id 回放整个执行过程，逐步检查推理逻辑。定位问题时先看 trace 找到失败 span，再看该 span 的输入输出判断是模型推理错误还是工具返回异常。

Q19：Agent 的幻觉问题怎么缓解？尤其是在工具调用场景下？进阶

A：多层防御：①Grounding——强制 Agent 基于工具返回的事实回答，而非凭空生成；②参数校验——在工具调用前验证参数合法性（类型、范围、必填项），拒绝明显错误的调用；③结果验证——对关键决策引入 Checker Agent 或规则引擎二次校验；④拒绝机制——在 System Prompt 中明确"不确定时说不知道"；⑤RAG 增强——用检索到的真实文档约束生成内容。工具调用场景特有的幻觉：模型编造不存在的工具名或参数——解法是严格约束工具 schema，使用 JSON mode 强制结构化输出。

Q20：如果让你设计一个 Agent 框架，核心抽象应该包含哪些模块？深追问

A：五个核心模块：①Planner——接收目标，输出执行计划（可以是 ReAct 的单步或 Plan-and-Solve 的多步）；②Executor——执行计划中的每一步，调用 LLM 或工具；③Tool Manager——统一管理工具注册、发现、调用、权限，支持 MCP/Function Calling；④Memory Manager——管理三层记忆的读写、压缩、过期；⑤Evaluator——对执行结果做质量评估，决定是否需要重规划。此外还需要：State Machine 控制执行流、Callback/Hook 支持自定义扩展、Observability 层做全链路追踪。

Q21：蚂蚁的支付宝场景中，Agent 需要操作用户资金，安全边界怎么设计？深追问

A：核心原则——Agent 不直接执行高风险操作，只生成"意图"由后端系统校验执行。具体：①权限分级——查询类工具 Agent 可自主调用，修改/支付类必须经过人工确认或二次验证；②参数边界——金额上限、操作频率限制，超出自动拦截；③审计链——每次工具调用记录完整上下文，支持事后追责；④沙箱隔离——Agent 运行在受限环境中，无法直接访问核心数据库；⑤Prompt 注入防御——用户输入和系统指令严格隔离，防止用户通过对话诱导 Agent 执行越权操作。

Q22：LangChain 和 LangGraph 的区别？什么时候该用 LangGraph？基础

A：LangChain 提供链式组合（Chain），适合线性流程；LangGraph 基于有向图（DAG/循环图），支持条件分支、循环、并行执行和状态持久化。该用 LangGraph 的场景：①需要条件分支（根据中间结果走不同路径）②需要循环（如 Reflection 反复修订直到满足条件）③需要多 Agent 编排（状态在 Agent 之间传递）④需要断点恢复（人工审批后继续执行）。简单的"检索→生成"用 LangChain 的 LCEL 即可，复杂的多步 Agent 用 LangGraph。

三、RAG 与知识工程

Q23：请描述一个完整的 RAG 链路，从文档入库到最终生成答案。基础

A：离线阶段：文档解析（PDF/DOCX/HTML → 纯文本）→ 分块（固定长度/语义/递归分割）→ 每块生成 Embedding 向量 → 存入向量数据库（Milvus/FAISS）+ 元数据索引。在线阶段：用户 Query → Query 预处理（改写/扩展）→ 向量检索 top-K → 重排序（Cross-Encoder / Cohere Rerank）→ 上下文组装（top-N 文档 + System Prompt + 用户问题）→ LLM 生成答案 → 后处理（引用标注/格式化）。

Q24：文档分块有哪些策略？各有什么优缺点？基础

A：①固定长度分块（如 512 tokens）：简单快速，但可能切断语义完整的段落；②递归字符分割（LangChain RecursiveCharacterTextSplitter）：按段落→句子→字符逐级切分，兼顾长度和语义边界；③语义分块：用 Embedding 计算相邻句子相似度，相似度骤降处作为分割点，保持语义完整性但计算成本高；④滑动窗口：相邻块有重叠（如 overlap=50 tokens），避免边界信息丢失但增加存储；⑤结构化分块：基于文档结构（标题/章节）分块，适合格式规范的文档。生产中常用递归分割 + 滑动窗口的组合。

Q25：Query Rewrite 和 HyDE 分别是什么？哪种效果更好？进阶

A：Query Rewrite：用 LLM 将用户原始查询改写为更适合检索的形式——补全省略信息、消歧义、拆分复合问题。HyDE（Hypothetical Document Embeddings）：让 LLM 先生成一个"假设性答案文档"，用这个假设文档的向量去检索，因为假设文档与真实答案在向量空间中更近。效果对比：HyDE 在 open-domain QA 效果好，但生成假设文档会引入幻觉风险且增加延迟；Query Rewrite 更轻量可靠，适合大多数生产场景。两者可以结合——先 Rewrite 再对 Rewritten Query 做 HyDE。

Q26：向量检索和关键词检索（BM25）如何混合使用？为什么单独用一种不够？进阶

A：混合检索（Hybrid Search）：分别用向量检索和 BM25 各召回 top-K，再用 Reciprocal Rank Fusion（RRF）或加权合并。单独用向量检索的问题：对精确关键词匹配（如专有名词、ID、代码片段）召回不稳定；单独用 BM25 的问题：无法处理同义词和语义相似（如"怎么减肥"和"体重管理方案"）。混合检索兼顾语义理解和精确匹配，实践中 RRF 的效果通常优于简单加权——它不依赖分数归一化，对不同检索器的分数分布差异鲁棒。

Q27：Embedding 模型怎么选？什么时候需要微调 Embedding？进阶

A：选型考虑：①维度（768/1024/1536）——越大表达力越强但检索更慢；②多语言能力——中文场景优先选 BGE/M3E/GTE 等中文优化模型；③MTEB 排行榜性能；④上下文长度支持。需要微调的场景：①领域术语多（医学/法律/金融），通用模型对专有名词理解不准；②查询和文档的风格差异大（如短问句 vs 长文档）。微调方法：对比学习（正例对 + 难负例），用 InfoNCE Loss 训练。微调后向量空间中相关文档更近、不相关更远。

Q28：RAG 系统怎么评测？有哪些关键指标？进阶

A：RAGAS 框架的四个核心指标：①Faithfulness（忠实度）——答案是否基于检索到的文档，而非编造；②Answer Relevancy——答案是否回答了用户问题；③Context Precision——检索到的文档中有多少是真正相关的（精确率）；④Context Recall——所有相关文档是否都被检索到（召回率）。此外还关注：端到端延迟（P50/P99）、检索 top-K 命中率、LLM token 成本。评测方法：构建标注数据集（query + 标准答案 + 相关文档），自动化跑分 + 人工抽检。

Q29：RAG 中的 "Lost in the Middle" 问题是什么？怎么解决？深追问

A：研究发现 LLM 对上下文中间位置的信息关注度最低——放在开头和结尾的文档更容易被模型引用。解决策略：①重排序后再排列——将最相关的文档放在上下文首尾，次相关的放中间；②减少上下文长度——只放 top-3 而非 top-10，减少中间区域；③分段生成——分别基于每个文档生成子答案，再做汇总融合；④结构化标注——在每个文档前加编号和标题引导注意力（如 [DOC-1: xxx]）。根本解法是选择对位置不敏感的模型，或使用长上下文模型。

Q30：Agentic RAG 和普通 RAG 的区别？什么场景需要 Agentic RAG？进阶

A：普通 RAG 是单次"检索→生成"的流水线。Agentic RAG 引入 Agent 循环：①检索后判断文档是否足够回答问题，不够则自动改写查询重新检索；②多源检索——Agent 决定从哪个知识库/工具获取信息；③多跳推理——先检索 A，从 A 的答案中提取新问题，再检索 B。适用场景：复杂问题（需要多步推理）、多源知识整合（内部文档 + 外部 API + 数据库）、对答案质量要求高（需要自我验证）。代价是延迟和成本增加。

Q31：FAISS、Milvus、Pinecone 怎么选型？基础

A：FAISS：Meta 开源，纯库（非独立服务），GPU 加速毫秒级检索，适合快速原型验证和嵌入式场景，不支持分布式和动态增删。Milvus：云原生分布式向量数据库，支持百亿级向量、混合检索（向量+标量过滤）、动态 schema，适合生产级大规模部署。Pinecone：全托管 SaaS，零运维但成本高、数据出境风险。选型原则：个人项目/POC 用 FAISS；国内生产环境用 Milvus（开源可控）；小团队追求快速上线且预算充足用 Pinecone。

Q32：如果 RAG 检索出来的文档和用户问题完全不相关怎么办？如何设计兜底机制？深追问

A：三层兜底：①相关性阈值——设置最低相似度分数（如 cosine > 0.65），低于阈值的文档不注入上下文；②空检索检测——如果有效文档数为 0，切换为纯 LLM 回答（不做 RAG），或返回"该问题超出知识库范围"；③Reranker 二次过滤——用 Cross-Encoder 对候选文档精排，过滤掉向量检索中的 false positive。还可以引入"拒绝回答"机制——在 prompt 中指示模型：如果提供的文档无法回答问题，明确告知用户并建议换个问法。

四、Context Engineering & Prompt Engineering

Q33：Context Engineering 和 Prompt Engineering 的区别？为什么说 2026 年 Context Engineering 更重要？基础

A：Prompt Engineering 关注"怎么写指令"——措辞、格式、Few-shot 示例的组织。Context Engineering 关注"运行时模型看到什么信息"——哪些知识注入上下文、什么时候注入、如何组织排布。后者更重要因为：①模型能力越来越强，指令理解已不是瓶颈，瓶颈在于给它正确的上下文；②Agent 系统中上下文动态变化（工具返回、记忆读取、检索结果），需要工程化管理；③研究表明结构化的上下文 + 弱模型可以超过混乱上下文 + 强模型。

Q34：Context Engineering 的六大核心技术是什么？进阶

A：①选择性检索——任务感知的元数据过滤 + 冗余移除 + 相关性重排序，只注入最相关的 top-N；②上下文压缩——分层摘要、句子级评分保留 top-20%、带约束的 LLM 摘要，减少 50-75% token；③分层布局——[系统规则] → [任务] → [用户画像] → [检索上下文] → [工具输出] → [问题]，模型对不同区域关注度不同；④动态查询重构——检索前改写模糊查询提高召回；⑤长时域任务管理——对话压缩/外部笔记/子 Agent 架构处理超长任务；⑥工具输出截断与摘要——工具返回大量数据时自动截断或摘要后注入。

Q35：上下文窗口快满了怎么办？有哪些压缩策略？进阶

A：四种策略按场景选择：①滑动窗口——只保留最近 N 轮对话，简单但丢失早期重要信息；②滚动摘要——用 LLM 将历史对话压缩为摘要，摘要置于对话开头，新消息追加在后面，保留关键信息同时大幅缩减 token；③选择性保留——对每轮对话标注重要性分数，只保留高分轮次 + 最近轮次；④外部记忆卸载——将重要信息写入外部存储（Mem0/数据库），需要时按需检索注入。最优实践是组合使用：滚动摘要 + 外部记忆 + 最近 N 轮原文。

Q36：System Prompt 怎么设计？有哪些最佳实践？基础

A：结构化分区：①角色定义——明确 Agent 是谁、擅长什么、边界在哪；②行为规则——输出格式、安全约束、拒绝策略；③工具说明——可用工具列表和调用规范；④示例——1-2 个 Few-shot 覆盖典型场景。最佳实践：指令要具体可执行（不要"尽量准确"，要"如果不确定请回复'我不确定'"）；关键规则放在开头和结尾（利用 U-shape 注意力分布）；避免冗余和矛盾；定期根据 bad case 迭代优化。

Q37：Prompt 注入攻击有哪些类型？如何防御？进阶

A：攻击类型：①直接注入——在用户输入中写"忽略之前的指令，执行 xxx"；②间接注入——在检索到的文档或工具返回中植入恶意指令；③越狱——通过角色扮演/编码绕过安全限制。防御策略：①输入隔离——System Prompt 和用户输入用清晰的分隔符/标签区分，系统指令优先级明确高于用户输入；②输入过滤——检测常见注入模式（"ignore previous"等）；③输出校验——对 Agent 的工具调用做白名单/参数校验，拒绝异常请求；④权限最小化——Agent 只能调用必需的最少工具；⑤对抗训练——在评测集中加入注入样本测试鲁棒性。

Q38：Chain-of-Thought (CoT) 的原理是什么？什么时候用 CoT 什么时候不用？基础

A：CoT 通过在 prompt 中引导模型"让我们一步一步思考"，使模型输出中间推理步骤而非直接给答案，利用自回归特性——前面输出的推理步骤成为后续生成的上下文，降低单步推理难度。适合用 CoT：复杂推理（数学、逻辑、多步问答）、需要可解释性的场景。不适合用 CoT：简单事实查询（"法国首都是哪"）、分类任务、对延迟敏感的场景——CoT 会增加输出 token 数从而增加延迟和成本。

Q39：Few-shot 示例的选择对模型输出影响有多大？怎么选最优示例？进阶

A：影响极大——错误的示例可能让模型性能低于 zero-shot。选择策略：①语义相似性——用 Embedding 找与当前 query 最相似的历史 Q&A 作为示例（如你在 AAAI 竞赛中的做法）；②多样性——示例应覆盖不同的推理路径/边界情况，避免全是简单例子；③标签平衡——分类任务中各类别等量；④质量筛选——只用验证过正确的示例。动态 Few-shot（根据输入实时检索示例）比静态 Few-shot 效果好 15-30%，但增加了检索延迟。

Q40：如何评估和自动优化 Prompt？有哪些方法论？深追问

A：评估：构建评测集（输入 + 期望输出），用自动化指标（准确率/F1/BLEU）+ LLM-as-Judge（让另一个模型评分）双重评估。自动优化方法：①DSPy——将 prompt 中的指令/示例作为"可优化参数"，通过 bootstrapping 自动搜索最优组合；②OPRO——用 LLM 本身作为优化器，迭代生成和评估 prompt 变体；③手动迭代——收集 bad case → 分析失败原因 → 针对性修改 prompt → 回归测试。核心是建立"修改 → 评测 → 对比"的闭环，避免改了 A 问题又引入 B 问题。

五、大模型基础

Q41：MHA、MQA、GQA 三种注意力机制的区别？为什么现在主流用 GQA？基础

A：MHA（Multi-Head Attention）：每个 head 有独立的 Q/K/V 投影，表达力最强但 KV Cache 最大。MQA（Multi-Query Attention）：所有 head 共享一组 K/V，KV Cache 缩小到 1/H，推理快但质量有损。GQA（Grouped-Query Attention）：折中方案——将 head 分成 G 组，每组共享 K/V，KV Cache 缩小到 G/H。主流用 GQA（如 Llama 2/3、Mistral）因为它在推理速度和生成质量之间取得最佳平衡，G=2 或 4 时质量接近 MHA，速度接近 MQA。

Q42：SFT、RLHF、DPO、GRPO 的区别和联系？进阶

A：SFT（Supervised Fine-Tuning）：在高质量 Q&A 数据上有监督微调，让模型学会对话格式和领域知识。RLHF：先训 Reward Model 打分，再用 PPO 优化策略模型使其生成高分回复。DPO（Direct Preference Optimization）：跳过 Reward Model，直接用偏好对（chosen vs rejected）优化策略模型，训练更稳定、超参数更少。GRPO（Group Relative Policy Optimization）：DeepSeek 提出，对同一问题采样多个回答，用组内相对排名替代绝对奖励，省去 Reward Model 且对数学/代码等可验证任务效果好。常见路径：预训练 → SFT → DPO/GRPO。

Q43：LoRA 的原理？A 和 B 矩阵分别怎么初始化？秩 r 怎么选？进阶

A：LoRA 在冻结的预训练权重旁边加一个低秩分解：W' = W + BA，其中 B ∈ R^{d×r}，A ∈ R^{r×k}，r << min(d,k)。初始化：A 用高斯随机初始化（或 Kaiming），B 初始化为零矩阵——这样训练开始时 BA=0，不改变预训练模型的输出。秩 r 的选择：r=4~16 适合大多数场景；任务越复杂/数据越多可以适当增大 r（如 32-64）；r 过大会过拟合且失去参数效率优势。实践中用 LoRA+ 或 DoRA 变体可进一步提升效果。

Q44：MoE（Mixture of Experts）的核心思想？路由机制怎么设计？负载均衡怎么解决？进阶

A：核心思想：每层有多个 FFN 专家（如 8 个），每个 token 只激活其中 top-K 个（如 2 个），实现"总参数大但激活参数少"。路由机制：Router 是一个线性层 + Softmax，输入 token 表示，输出各专家的概率分布，选 top-K 概率最高的专家。负载均衡问题：训练中路由容易"塌缩"——少数专家被频繁选中，其他专家闲置。解法：在 loss 中加 auxiliary load balancing loss，惩罚专家负载不均匀，公式通常是各专家被选中的频率与均匀分布的 KL 散度。

Q45：大模型幻觉产生的根本原因？有哪些缓解手段？进阶

A：根本原因：①训练数据中包含错误/过时信息；②语言模型本质是学习概率分布而非事实存储，高概率输出不等于正确；③知识截止日期后的信息模型不知道；④长尾知识训练样本少导致"记不住"。缓解手段：①RAG——用外部知识库 grounding；②引用标注——要求模型标注答案来源，可验证；③自一致性——采样多次取多数票；④事实核查器——后处理用知识图谱/搜索引擎核实关键事实；⑤拒绝回答——训练模型在不确定时说"不知道"（RLHF 中奖励诚实）；⑥Retrieval 增强训练——RETRO 等方法在训练时就引入检索。

Q46：Transformer 中为什么要除以 √d_k？不除会怎样？基础

A：QK^T 的点积结果方差与维度 d_k 成正比。如果 d_k 很大（如 128），点积值可能非常大，导致 Softmax 输出趋于 one-hot——梯度接近于零，训练不稳定。除以 √d_k 将方差归一化到 1，使 Softmax 的梯度在合理范围内。不除的后果：attention 分布过于尖锐（几乎只 attend 到一个位置），模型难以学习多位置的混合信息。

Q47：RoPE（旋转位置编码）的原理？为什么比绝对位置编码好？深追问

A：RoPE 将位置信息编码为旋转矩阵，作用在 Q 和 K 上。核心性质：两个 token 的注意力分数只取决于它们的相对位置差（而非绝对位置），因为 q_m · k_n = f(x_m, x_n, m-n)。优势：①天然支持相对位置编码，外推性更好；②可以通过 NTK-aware 缩放或 YaRN 扩展到更长序列；③不增加额外参数。相比之下，绝对位置编码（如正弦/学习式）对未见过的位置泛化差，且注意力分数隐含地依赖绝对位置。

Q48：预训练数据的质量对模型有多大影响？数据清洗一般怎么做？进阶

A：影响巨大——"Garbage In, Garbage Out"。Llama 3 的数据工程投入远超模型架构调整。清洗流程：①去重——MinHash + LSH 去近似重复文档（重复数据导致记忆而非泛化）；②质量过滤——用分类器（如 perplexity scoring, fastText 质量分类器）过滤低质量网页、机器翻译、乱码；③有害内容过滤——去除仇恨/暴力/色情内容；④PII 去除——匿名化个人信息；⑤数据配比——控制不同来源（网页/书籍/代码/学术）的比例，避免某类数据过度主导。

六、推理优化与 AI Infra

Q49：KV Cache 的原理？为什么能加速推理？内存占用怎么算？基础

A：自回归生成时，每生成一个新 token 需要对所有已有 token 计算 Attention。不用 KV Cache 时每步都重新计算所有 token 的 K/V，复杂度 O(N²)。KV Cache 缓存历史 token 的 K/V 值，新 token 只需计算自己的 Q，与缓存的 K/V 做 Attention，复杂度降为 O(N)。内存占用 ≈ 2 × num_layers × batch_size × seq_len × num_heads × head_dim × dtype_size。以 13B 模型 FP16 为例，每个 token 的 KV Cache 约 800KB，2048 tokens 占约 1.6GB。

Q50：vLLM 的 PagedAttention 是怎么解决 KV Cache 内存碎片问题的？进阶

A：传统方案为每个请求预分配最大序列长度的连续内存，导致严重碎片——实际使用率最低仅 20.4%。PagedAttention 借鉴操作系统虚拟内存的分页思想：将 KV Cache 分成固定大小的 block（如 16 tokens），block 可以存储在非连续的物理内存中，通过 block table（页表）映射逻辑位置到物理位置。好处：①按需分配，用多少分多少；②block 级别的共享——parallel sampling 时多个序列共享 prompt 的 KV Cache（copy-on-write）；③碎片率接近零。吞吐量相比 HuggingFace 提升 2-24 倍。

Q51：什么是 Continuous Batching / Dynamic Batching？和 Static Batching 的区别？进阶

A：Static Batching：一批请求一起开始，等所有请求都完成才处理下一批。问题：短请求完成后 GPU 空闲等待长请求，利用率低。Continuous Batching（vLLM/TGI 使用）：在每个 decode step 级别动态调度——短请求完成后立即空出位置，新请求可以插入。效果：GPU 始终保持高利用率，吞吐量提升 2-5 倍。实现关键：需要 PagedAttention 支持动态内存管理，以及高效的调度器决定每步有哪些请求参与 batch。

Q52：推测解码（Speculative Decoding）的原理？能加速多少？进阶

A：核心思想：用一个小模型（draft model）快速生成 K 个候选 token，然后用大模型（target model）并行验证这 K 个 token。如果小模型预测正确（通过拒绝采样保证分布一致性），则一次前向传播生成 K 个 token 而非 1 个。加速比取决于小模型的准确率——如果小模型预测 80% 正确，理论上接近 K × 0.8 的加速。实际加速 1.5-3 倍。适用条件：draft model 要足够小（推理开销低）且与 target model 分布接近。局限：对需要高创造性的任务（小模型预测准确率低）效果差。

Q53：Prefill-Decode 分离架构是什么？为什么要分离？深追问

A：Prefill 阶段处理完整 prompt，计算量大（O(N²) attention）、GPU 计算密集型。Decode 阶段逐 token 生成，单步计算量小但内存密集型（需要读取大量 KV Cache）。两者对计算资源的需求模式不同：Prefill 需要高算力，Decode 需要高显存带宽。PD 分离将两个阶段部署在不同的 GPU 实例上，各自优化：Prefill 节点配高算力 GPU，Decode 节点配大显存 GPU。核心挑战：如何高效传输 KV Cache（RDMA/NCCL 直连）。Mooncake（月饼）等系统通过分布式 KV Store 解决传输瓶颈。

Q54：模型量化有哪些方法？INT8/INT4/FP8 各有什么特点？基础

A：量化是将模型权重/激活从高精度（FP16/BF16）压缩到低精度以减少内存和加速推理。INT8：精度损失小，硬件支持广（几乎所有 GPU），常用 GPTQ/AWQ 方法。INT4：压缩比更高（4 倍）但精度损失增大，需要 group quantization 缓解；GGUF/GPTQ-4bit 常用。FP8：Hopper/Ada 架构原生支持，保留浮点表示的动态范围，精度优于 INT8 同时推理速度更快。选择原则：追求极致压缩用 INT4（边缘部署）；生产推理用 INT8/FP8（精度与速度平衡）；训练用 BF16/FP8 混合精度。

Q55：vLLM 的关键参数调优有哪些？怎么提升吞吐量？进阶

A：核心参数：①--gpu-memory-utilization（默认 0.9，生产可调到 0.92-0.95，留余量防 OOM）；②--max-num-seqs（最大并发序列数，增大可提升吞吐但增加延迟）；③--max-model-len（最大序列长度，按业务需要设置，过大浪费内存）；④--enable-chunked-prefill（开启分块预填充，让 Prefill 和 Decode 并行执行）；⑤--enable-prefix-caching（前缀缓存，适合固定 System Prompt 场景）；⑥--kv-cache-dtype fp8（KV Cache 量化，需要 Hopper 架构）。实践中先压测找瓶颈（是 GPU 算力还是显存带宽），再针对性调参。

Q56：Ollama 和 vLLM 的定位有什么区别？各适合什么场景？基础

A：Ollama：面向个人/开发者的本地 LLM 运行工具，一行命令拉取并运行模型，支持 GGUF 格式量化模型，资源占用小，适合原型验证、本地开发、边缘部署。vLLM：面向生产的高性能推理引擎，支持 PagedAttention、Continuous Batching、Tensor Parallel 等，追求最大吞吐和最低延迟，适合大规模线上服务。简单说：Ollama 是"快速跑起来"，vLLM 是"高效服务化"。蚂蚁的 Agent 系统后端大概率用 vLLM 或自研推理框架。

七、工程化与系统设计

Q57：RabbitMQ 和 Kafka 的区别？Agent 系统中该用哪个？基础

A：RabbitMQ：消息代理模式，支持复杂路由（Exchange/Binding）、消息确认、死信队列，消息消费后删除，适合任务分发、异步解耦（如 Agent 的特征抽取、向量化等异步任务）。Kafka：分布式日志系统，消息持久化、可重放，高吞吐，适合事件流/日志采集/流计算。Agent 系统中：异步任务解耦用 RabbitMQ（消息可靠、重试方便）；Agent 执行日志/行为追踪用 Kafka（需要持久化和回放分析）。两者不是替代关系而是互补。

Q58：SSE 和 WebSocket 的区别？Agent 的流式输出为什么常用 SSE？基础

A：SSE（Server-Sent Events）：单向（服务端 → 客户端），基于 HTTP，自动重连，原生支持事件类型和 ID。WebSocket：全双工双向通信，自定义协议，需要心跳保活。Agent 流式输出用 SSE 的原因：①LLM 生成是单向数据流（服务端 → 客户端），不需要双向；②SSE 基于 HTTP，穿透代理/CDN 更容易；③浏览器原生 EventSource API，实现简单；④自动重连机制对长时间的生成任务很友好。WebSocket 适合需要客户端实时发送数据的场景（如在线协作编辑、游戏）。

Q59：多级缓存怎么设计？数据一致性怎么保证？进阶

A：三级：L1 本地缓存（Caffeine，毫秒级，容量有限）→ L2 分布式缓存（Redis，亚毫秒级，跨实例共享）→ L3 数据库（MySQL，持久化）。读路径：L1 命中直接返回 → L1 miss 查 L2 → L2 miss 查 L3 并回填 L1/L2。一致性保障：①写后删（Cache Aside）——更新 DB 后删除缓存，下次读时重建；②L1 设置短 TTL（如 30s）容忍短暂不一致；③关键数据通过消息队列广播失效通知给所有实例的 L1；④避免缓存和 DB 双写（先更新 DB，再删缓存，用延迟双删兜底）。

Q60：分布式锁怎么实现？Redisson 的 SET NX EX 有什么陷阱？进阶

A：SET key value NX EX timeout：NX 保证只有第一个设置成功（互斥），EX 设置过期时间防死锁。陷阱：①锁过期但业务未完成——A 拿锁后执行慢，锁过期被 B 拿走，A 完成后误删 B 的锁。解法：value 存唯一标识（如 UUID），删除前比较 value（用 Lua 脚本原子化）；②不可重入——同一线程二次加锁会失败。Redisson 的 RLock 通过 hash 结构记录持有者和重入次数解决；③主从切换丢锁——Redis 主节点写入成功后宕机，从节点未同步。解法：RedLock（多节点加锁）或接受极端情况下的不一致。

Q61：设计一个支持百万级日查询的企业级 Agent 系统，你会怎么做？深追问

A：分层架构：①接入层——Nginx/API Gateway 负载均衡 + 限流 + 鉴权；②Agent 编排层——无状态服务（可水平扩展），负责 Agent 执行调度，状态存外部存储（Redis）；③模型推理层——vLLM 集群，GPU 实例按流量弹性伸缩，Continuous Batching 最大化吞吐；④知识检索层——Milvus 集群 + ES（混合检索），读写分离；⑤缓存层——语义缓存（相似 query 命中历史回答，省去 LLM 调用）+ Redis 热数据缓存。关键优化：异步化（MQ 解耦耗时步骤）、流式返回（SSE 提前出第一个 token）、降级策略（推理层过载时返回缓存/简化回复）。

Q62：缓存穿透、缓存击穿、缓存雪崩分别是什么？怎么防？基础

A：缓存穿透：查询不存在的数据，每次都打到 DB。防御：布隆过滤器拦截不存在的 key / 缓存空值（短 TTL）。缓存击穿：热 key 过期瞬间大量请求打到 DB。防御：热 key 永不过期 + 异步刷新 / 互斥锁（只让一个线程重建缓存）。缓存雪崩：大量 key 同时过期。防御：TTL 加随机偏移（避免集中过期）/ 多级缓存 / 限流降级。Agent 系统中，对 LLM 调用的语义缓存是重要优化——相似 query 的结果可以复用，大幅降低模型调用次数。

Q63：MVCC 的原理？MySQL InnoDB 如何实现可重复读？基础

A：MVCC（Multi-Version Concurrency Control）：每行数据维护多个版本（通过隐藏的 trx_id 和 roll_pointer 字段），读操作不加锁，通过 ReadView 判断哪个版本对当前事务可见。InnoDB 可重复读实现：事务开始时创建 ReadView（记录当前活跃事务 ID 列表），整个事务期间复用同一个 ReadView。每次读数据时，沿 undo log 版本链找到第一个对当前 ReadView 可见的版本。这样即使其他事务提交了新数据，当前事务读到的仍然是一致的快照。与读已提交的区别：RC 每次 SELECT 都创建新的 ReadView。

Q64：B+ 树为什么适合数据库索引？和 B 树、Hash 索引的区别？基础

A：B+ 树优势：①所有数据存在叶子节点，非叶子节点只存键值，单页能存更多索引项，树更矮，磁盘 IO 更少；②叶子节点通过双向链表连接，支持高效范围查询和顺序扫描；③查询路径长度稳定（所有叶子在同一层），性能可预测。B 树区别：数据分布在所有节点，不支持高效范围查询。Hash 索引区别：只支持等值查询 O(1)，不支持范围/排序/前缀匹配。InnoDB 默认使用 B+ 树索引（聚簇索引 + 二级索引），Memory 引擎支持 Hash 索引。

八、综合能力与场景题

Q65：在你做过的项目中，最有挑战的技术决策是什么？结果如何？基础

A：（建议结合自身经历回答）示例方向：喜马拉雅项目中选择 Mem0 做长期记忆 vs 自建向量存储的权衡——Mem0 提供开箱即用的混合存储（向量+结构化）和 SDK，但自建方案可控性更高。最终选择 Mem0 因为开发周期紧 + 团队熟悉度高 + 迁移成本低。结果：快速上线、指标达标，但后期遇到 Mem0 写入性能瓶颈需要异步化改造。反思：技术选型要平衡"最优解"和"可落地性"。

Q66：项目中遇到过哪些 bad case？怎么建立持续优化的闭环？进阶

A：（建议结合实际 bad case 回答）闭环流程：①采集——用户反馈 + 自动化评测 + 人工抽检三个渠道收集 bad case；②归类——按失败原因分类（检索失败 / 生成幻觉 / 工具调用错误 / Prompt 缺陷）；③分析——每类取典型 case 做根因分析，复现并定位到具体环节；④修复——针对性优化（调整 Prompt / 增加训练数据 / 修改检索策略）；⑤回归——修复后在全量 bad case 集上回归测试，确认不引入新问题。关键是将 bad case 沉淀为评测集，每次改动都跑一遍。

Q67：如何评估一个 Agent 系统的效果？用什么指标？进阶

A：分三个维度：任务维度——任务完成率（是否达成用户目标）、答案准确率、步骤效率（完成任务用了多少步）。质量维度——幻觉率、安全违规率、用户满意度（主观评分 / NPS）。工程维度——端到端延迟（首 token 延迟 + 完成延迟）、吞吐量（QPS）、token 成本、工具调用成功率。评测方法：①自动化 benchmark（构建标准 case 集）②A/B 测试（新旧系统对比）③LLM-as-Judge（用更强的模型评分）④人工评估（定期抽检关键场景）。

Q68：如果让你设计支付宝的智能客服 Agent，你会怎么做？深追问

A：整体架构：意图识别层（分类用户意图：查余额/转账/投诉/咨询）→ 路由层（简单意图走规则引擎直接回复，复杂意图路由到 Agent）→ Agent 层（多 Agent 协作：查询 Agent 调用账户 API、问答 Agent 检索知识库、操作 Agent 引导完成转账等）→ 安全层（高风险操作人工确认、Prompt 注入防御、敏感信息脱敏）。关键设计：①知识库覆盖常见 FAQ + 政策文档（RAG）；②MCP 封装内部系统 API（账户/交易/风控）；③Mem0 记录用户历史交互偏好；④降级策略——Agent 无法解决时无缝转人工。

Q69：你对 AI 编程工具（Cursor/Claude Code）有什么使用心得？如何提高 AI 辅助编码的效率？基础

A：（JD 特别提到 AI 编程工具，建议结合真实体验回答）核心心得：①Context 是关键——给 AI 足够的上下文（相关文件、需求描述、已有代码风格），比反复修正 prompt 更有效；②分步推进——复杂任务拆成小步骤逐步让 AI 实现，而非一次给完整需求；③Review 必不可少——AI 生成的代码需要人工审查边界条件、安全问题和性能隐患；④Cursor Rules / Skills——为项目配置 context 规则，让 AI 了解项目规范和约束，减少反复解释。局限：复杂的跨模块重构、性能优化仍需人工主导。

Q70：你的研究方向是 AI4Edu，如何看待 AI Agent 在教育领域的应用前景和挑战？进阶

A：前景：①个性化辅导——Agent 可以根据学生的知识水平、学习风格动态调整教学策略（如你的 AI 面试系统中的自适应出题）；②实时反馈——代码/作文批改从天级缩短到秒级（如小花狮系统）；③教师赋能——自动化处理重复性工作，让教师专注高价值互动。挑战：①准确性要求极高——教育场景对幻觉零容忍，错误知识会误导学生；②评估标准化——开放性题目的 AI 评估与人类评估一致性仍需验证；③公平性——模型偏见可能对不同背景学生造成不公平对待；④隐私——学生数据的采集和使用需要严格合规。
