TCP 握手是三次，挥手是四次为什么？

计算机网络

三次握手流程：① 客户端 SYN（seq=x）→ ② 服务端 SYN+ACK（seq=y, ack=x+1）→ ③ 客户端 ACK（ack=y+1）

为什么是三次：

  • 防止历史连接：若只用两次握手，旧 SYN 可能因网络延迟迟到。服务端收到后误以为新连接，回复 SYN+ACK 并分配资源。客户端发现是旧连接会发 RST，但服务端已浪费资源。
  • 同步序列号：双方需确认对方的 ISN（初始序列号），用于后续重传、去重、按序组装。
  • 避免资源浪费：两次握手时，若客户端 SYN+ACK 丢失，服务端无法确认客户端已收到，会一直等待并重发，造成资源浪费。

四次挥手流程：① 客户端 FIN → ② 服务端 ACK → ③ 服务端 FIN → ④ 客户端 ACK（进入 TIME_WAIT）

为什么是四次：TCP 全双工，双方需各自关闭发送方向。被动方收到 FIN 时可能还有数据未发完，必须先 ACK 确认收到关闭请求，等数据发完后再发 FIN，因此 ACK 和 FIN 必须分开发送。

TCP 三次握手和四次挥手的区别，为什么少一次？

计算机网络

握手少一次的原因：服务端收到 SYN 后，可以立即回复 SYN+ACK，将「确认」和「自己的 SYN」合并为一条报文，无需等待。

挥手多一次的原因：被动方收到 FIN 时，可能还有数据未发送完，必须先 ACK 确认收到关闭请求，等数据发完后再发 FIN，因此 ACK 和 FIN 必须分开发送。

TCP 与 UDP 区别

计算机网络

TCP：面向连接、可靠传输（确认、重传、拥塞控制、流量控制）、有序、面向字节流。头部 20 字节起，开销大。

UDP：无连接、不可靠、无序、面向报文。头部 8 字节，开销小。

应用场景：TCP 适合 HTTP、FTP、邮件等需可靠传输的场景；UDP 适合视频直播、游戏、DNS 查询、广播等对实时性要求高、可容忍少量丢包的场景。

延展：QUIC 基于 UDP 实现可靠传输，解决 TCP 队头阻塞；TCP 的拥塞控制有慢启动、拥塞避免、快重传、快恢复等算法。

HTTP 与 HTTPS 的区别

计算机网络

核心区别：HTTP 明文传输；HTTPS = HTTP + SSL/TLS，加密传输。端口 80 vs 443。

HTTP 三大问题：窃听（明文可被中间人看到）、篡改（无完整性校验）、冒充（无身份验证）。

HTTPS 加密流程：① 客户端：加密套件 + 随机数 A；② 服务端：选择套件 + 证书 + 随机数 B；③ 客户端：校验证书、用公钥加密随机数 C；④ 双方用 A+B+C 生成会话密钥；⑤ 后续对称加密传输。

混合加密：非对称加密用于密钥交换（安全但慢）；对称加密用于数据传输（快）。

延展：证书链验证（服务器证书→中间 CA→根 CA）；TLS 1.3 简化握手到 1 RTT；HSTS 强制 HTTPS。

HTTP/1.0 到 HTTP/3 演进

计算机网络

HTTP/1.0：一个请求一次 TCP 连接，成本高；无 Host 头。

HTTP/1.1：Keep-Alive 长连接、Host 头、虚拟主机、分块传输。但响应按序返回，队头阻塞；文本协议头重复；需多连接并行拉资源。

HTTP/2：多路复用（一连接多请求/响应并行）、头部压缩 HPACK、二进制分帧、服务端推送。底层仍 TCP，丢包时整连接阻塞（队头阻塞）。

HTTP/3：基于 QUIC（UDP），每个 stream 独立，某 stream 丢包不影响其他；0-RTT 建连；内置 TLS 1.3。

延展：队头阻塞是 TCP 特性；QUIC 在用户态实现拥塞控制，可快速迭代。

打开网页发生了什么

计算机网络

① 解析 URL：拆出协议、域名、端口、路径。

② 缓存判断：浏览器缓存 → 系统缓存 → 路由器缓存 → 运营商缓存，命中则直接返回 IP。

③ DNS 解析：未命中则本地 DNS 迭代查询根→TLD→权威 DNS，得到 IP。

④ 获取 MAC：同子网则 ARP 获取目标 MAC；不同子网则 ARP 获取网关 MAC。

⑤ TCP 三次握手：建立连接。

⑥ TLS 握手：HTTPS 时协商密钥、校验证书。

⑦ HTTP 请求：发送请求行、头、体；服务器处理并返回状态码、头、体。

⑧ 解析渲染：解析 HTML 构建 DOM、解析 CSS 构建 CSSOM、合成渲染树、布局、绘制。遇到 script 可能阻塞。

延展：CDN 加速、HTTP/2 多路复用、资源预加载、关键渲染路径优化。

TLS 握手过程

计算机网络

① 客户端：支持的加密方式、协议版本、随机数 a、公开材料 A；② 服务端：选择加密方式、证书（含公钥）、随机数 b、公开材料 B；③ 客户端：校验证书、域名、有效期；④ 双方：a+B/b+A 算出会话密钥 K；⑤ 后续用 K 对称加密传输。

DNS 解析过程

计算机网络

域名层级：根域名（.）→ 顶级域名（.com/.cn）→ 二级域名（.baidu.com）→ 三级域名（www.baidu.com）。

解析流程：① 查浏览器/系统/路由器缓存；② 无则请求本地 DNS（递归查询）；③ 本地 DNS 迭代：根域名服务器 → 顶级域名服务器 → 权威 DNS 服务器，逐级返回下一级服务器地址；④ 最终得到 IP 返回。

递归 vs 迭代：客户端→本地 DNS 为递归（客户端只发一次请求）；本地 DNS→根/TLD/权威 为迭代（服务器逐级查询）。

延展：DNS 记录类型 A（IPv4）、AAAA（IPv6）、CNAME（别名）、MX（邮件）；DNS 劫持、污染；HTTPDNS 绕过运营商 DNS 防劫持。

跨域与 CORS

计算机网络

同源策略：协议、域名、端口三者相同才同源。不同源则跨域，浏览器限制脚本访问跨域资源。

解决方案：① CORS：服务端设置 Access-Control-Allow-Origin 等响应头；预检请求（OPTIONS）用于非简单请求；② JSONP：利用 script 标签无跨域限制，但仅 GET、安全性差；③ 代理：同源服务器转发请求；④ postMessage：跨窗口通信。

延展：简单请求（GET/POST/HEAD + 简单头）不发预检；复杂请求先 OPTIONS 预检再实际请求。

CDN 原理

计算机网络

核心思想：将静态资源缓存到离用户更近的节点，就近访问，减少源站压力和延迟。

流程：用户请求 → 本地 DNS 解析（CNAME 指向 CDN 调度器）→ 调度器根据用户 IP、节点负载返回最优边缘节点 IP → 用户访问边缘节点；边缘节点有则直接返回，无则回源拉取并缓存。

延展：边缘节点分布全球；缓存策略（强缓存、协商缓存）；预热、刷新、 purge；DDoS 防护；视频 CDN 多级分发。

Netty 与 Tomcat 区别

Netty

定位：Tomcat 是 Web 容器（Servlet+HTTP）；Netty 是网络通信框架。

协议：Tomcat 主要 HTTP/HTTPS；Netty 适合长连接、自定义协议（IM、游戏、TCP 私有协议）。

线程模型：Tomcat 一请求一线程；Netty 是 Reactor 模型，BossGroup 接收、WorkerGroup 处理 IO，多请求共享少量线程。

SSE vs WebSocket

Netty / 网络

选 SSE：服务端单向推送、流式输出；自带重连、Last-Event-ID 续传；更轻量。

选 WebSocket：双向、强交互、频繁通信。

SSE 真流式：Nginx 需 proxy_buffering off；响应头 Content-Type: event-stream；Connection: keep-alive。

ByteBuf 优化

Netty

读写指针分离：readerIndex/writerIndex，无需 flip/clear。

自动扩容：超容量自动扩展。

池化：PooledByteBufAllocator 对象池复用。

零拷贝：减少用户态/内核态复制；mmap、sendfile。

进程与线程的区别及通信方式

操作系统

区别：进程是资源分配单位，拥有独立地址空间、文件句柄等；线程是 CPU 调度单位，共享进程资源（堆、方法区），独有栈、PC、寄存器。进程创建/切换开销大（需切换页表、TLB 刷新），线程开销小。一个线程崩溃可能导致整个进程死亡。

进程间通信（IPC）：管道（匿名/命名）、消息队列、共享内存（最快，需配合信号量）、信号量、信号、套接字（跨主机）。

线程间通信：共享地址空间，可直接访问全局变量；同步用互斥锁、条件变量、信号量等。

延展：协程：用户态轻量级线程，由用户调度；内核线程 vs 用户线程（1:1、N:1、M:N 模型）。

进程调度算法

操作系统

FCFS（先来先服务）：按到达顺序调度，非抢占。简单但护航效应（长作业阻塞短作业）。

SJF（短作业优先）：选估计运行时间最短的进程。平均等待时间最优，但长作业可能饥饿，且需预知运行时间。

RR（时间片轮转）：抢占式，按 FCFS 排队，每进程执行一个时间片后回队尾。公平、响应快；时间片过大退化为 FCFS，过小增加上下文切换。一般 10~100ms。

延展：多级反馈队列：结合 FCFS、RR、优先级；Linux CFS 调度器基于红黑树、虚拟运行时间。

虚拟内存与页面置换

操作系统

虚拟内存：将物理内存和磁盘结合，为进程提供比物理内存更大的连续地址空间。通过页表映射虚拟地址到物理地址。

页面置换：缺页时若物理页已满，需淘汰一页。OPT（最优，不可实现）→ FIFO（简单，可能 Belady 异常）→ LRU（基于局部性，需硬件支持）→ 时钟算法（改进 FIFO，用使用位）→ 改进型时钟（使用位+修改位，优先换未修改的）。

延展：缺页时从磁盘调入约 25ms，远超其他操作，降低缺页率至关重要；TLB 加速地址转换。

死锁条件与预防

操作系统

四个必要条件：① 互斥：资源不能共享；② 持有并等待：持有资源的同时等待其他资源；③ 不可剥夺：资源只能主动释放；④ 环路等待：存在进程链形成环形等待，每个进程都在等下一个进程持有的资源。

预防：破坏其一即可。破坏环路等待：资源有序分配，按编号顺序申请；破坏持有并等待：一次性申请所有资源或申请前释放已有资源。

避免：银行家算法动态检测安全状态；加锁时限（超时释放重试）；锁顺序统一。

select / poll / epoll

操作系统

select：最多 1024 个 fd；每次调用拷贝 fd 集合到内核；O(n) 轮询所有 fd；可移植性好。

poll：无 fd 数量限制；仍需每次拷贝；同样 O(n) 轮询。

epoll：无 fd 限制；内核维护红黑树+就绪链表，无需每次拷贝；只关心就绪 fd，效率与连接总数无关；支持 ET/LT 两种触发模式。适合高并发。

延展：epoll 是 Linux 特有；epoll_create 创建、epoll_ctl 注册、epoll_wait 等待；ET 边缘触发需一次读完，否则可能丢事件。

同步阻塞、同步非阻塞、异步阻塞、异步非阻塞的区别

操作系统 / 并发

同步/异步：调用方是否等待结果。同步：调用后阻塞直到拿到结果；异步：调用后立即返回，结果通过回调/通知获取。

阻塞/非阻塞：等待时线程是否挂起。阻塞：线程挂起；非阻塞：线程可做其他事。

  • 同步阻塞 (BIO)：read() 一直等数据，线程阻塞。典型：传统 Socket。一连接一线程。
  • 同步非阻塞 (NIO 轮询)：read() 立即返回，无数据则轮询。CPU 空转。
  • 异步阻塞：少见，如 select 等待多个 fd 但阻塞。
  • 异步非阻塞 (epoll/多路复用)：注册 fd 和回调，事件就绪时通知，无需轮询。高并发首选。

延展：Reactor 模式（单/多线程 Reactor）；Proactor 模式（异步 IO）；Java NIO 的 Selector 对应 epoll。

乐观锁和悲观锁的区别

并发

悲观锁：假设冲突多，先加锁再操作。synchronized、ReentrantLock、SELECT FOR UPDATE、行锁。

乐观锁：假设冲突少，无锁，通过版本号/CAS 检测冲突后重试。如：UPDATE ... SET version=version+1 WHERE version=?

选型：读多写少、冲突少用乐观锁（无锁开销）；写多、冲突多用悲观锁。

Redis 分布式锁

系统设计

基本实现：SET key value NX EX seconds 原子设置。value 用 UUID 或线程唯一标识，防止误删其他客户端的锁。

删除时：必须先判断 value 是否为自己，否则可能删别人的锁（如 A 的锁过期后 B 获取，A 此时删除会误删 B 的锁）。用 Lua 脚本保证「判断+删除」原子性。

锁续期：业务执行时间可能超过 TTL。Redisson 看门狗：未指定 leaseTime 时，每 1/3 过期时间自动续期，直到 unlock() 或 JVM 挂掉。

Redlock：多 Redis 实例（通常 5 个），超过半数加锁成功才认为成功，防止单点故障。加锁时需设置相同的 value 和超时时间。

volatile 关键字

Java 并发

内存可见性：写 volatile 立即刷主存，其他核缓存失效（MESI 协议）；禁止变量缓存到寄存器，强制读 Cache Line，避免编译器/JIT 优化导致一直用寄存器旧值。

禁止指令重排：在读写处插入内存屏障；volatile 写 release（禁止写前操作跑到写后）；volatile 读 acquire（禁止读后操作跑到读前）。典型 DCL 单例需 volatile 防止 instance 引用先可见但对象未初始化完成。

不保证原子性：i++ 分读-改-写三步，需 AtomicInteger 或 synchronized。

AQS 抽象队列同步器

Java 并发

AbstractQueuedSynchronizer，构建锁和同步组件的基础。内部 FIFO 队列 + state 变量。ReentrantLock、Semaphore、CountDownLatch 基于 AQS。state=0 无锁，CAS 修改 state 获取锁，失败则入队等待。

CAS 与 ABA 问题

Java 并发

CAS：Compare-And-Swap，内存值等于期望值才更新。乐观锁实现。

ABA：A→B→A，指针/引用类可能致命。解决：版本号机制，(引用, version) 一起 CAS。

高并发自旋：CPU 空转。解决：退避、LongAdder 分散热点。

ReentrantLock

Java 并发

可重入锁，基于 AQS。相比 synchronized：可中断、可超时、可公平锁、多 Condition。FairSync 排队；NonfairSync 直接 CAS 抢锁。Condition 支持多条件队列，精准唤醒。

synchronized 锁升级

Java 并发

四种状态：无锁(01) → 偏向锁(01) → 轻量级锁(00) → 重量级锁(11)，只能升级不能降级。

偏向锁：单线程反复进入同步块时，Mark Word 记录线程 ID，无竞争时几乎无开销。有 4 秒延迟启动。

轻量级锁：另一线程竞争时，在栈帧创建 Lock Record，CAS 将 Mark Word 指向 Lock Record；自旋获取，失败则升级。

重量级锁：自旋超过阈值（约 10 次）或多线程竞争，升级为 Monitor，Mark Word 指向 Monitor 对象，线程进入阻塞队列，依赖 OS 互斥量。

缓存雪崩、穿透、击穿的区别与解决方案

Redis / 缓存 · 大厂高频

三者区别：穿透（数据在缓存和 DB 都不存在，请求直打 DB，常见于非法 ID、恶意攻击）；击穿（热点 key 过期瞬间，大量并发同时打 DB）；雪崩（大量 key 同时失效或 Redis 宕机，请求集中打 DB）。

穿透：① 网关层参数校验（非法 ID 直接拦截）；② 布隆过滤器：False 一定不存在直接返回，True 可能存在放行；③ 缓存空对象，短 TTL 避免缓存污染。新增/修改时同步更新布隆过滤器。

击穿：SingleFlight：ConcurrentHashMap<key, CompletableFuture>，computeIfAbsent 保证同 key 只有第一个线程创建 Future 并查 DB，其余线程 get() 等待。Leader 查完回写 Redis 后 Future 完成，所有等待线程复用结果。

雪崩：① 随机 TTL：过期时间 + random(0, 20min)，避免同时失效；② 多级缓存；③ 限流熔断；④ Redis 主从/集群高可用。

Redis 热 Key 问题如何发现和解决？

Redis / 缓存 · 大厂高频

定义：某 Key 访问量远高于其他，如总 QPS 10000 时某 Key 达 7000，或对大 Hash/ZSet 频繁操作。

危害：单节点网卡打满、缓存击穿、DB 雪崩、Redis Cluster 分片不均衡。

发现：经验预测（秒杀商品 ID）；redis-cli --hotkeys；MONITOR+redis-faina 分析；客户端/代理层统计访问频率。

解决：① 本地缓存（Guava/Caffeine）分散请求；② 热 Key 拆分：key_001～key_015 分散到不同节点，读写时随机选；③ 读写分离+从节点扩容；④ 热 Key 永不过期+MQ 异步更新。

Redis RDB 与 AOF 持久化区别

Redis / 缓存

RDB：二进制快照，某一时刻完整数据。触发：save 900 1、BGSAVE。恢复快、文件小，可能丢失分钟级数据。

AOF：追加写命令，重启时重放。刷盘：always（每次）、everysec（默认）、no。数据安全高，恢复慢、文件易膨胀。

混合模式（4.0+）：RDB 做全量 + AOF 做增量，兼顾恢复速度与数据安全。

限流算法：漏桶 vs 令牌桶

Redis / 缓存 · 大厂高频

漏桶：请求进桶排队，以固定速率流出。无论请求多快，输出速率恒定。无法处理突发，输出非常平滑。

令牌桶：以恒定速率往桶里放令牌，请求取到令牌才通过。桶有容量，可应对短时突发（桶满时相当于漏桶）。

区别：令牌桶允许突发、漏桶强制平滑。令牌桶保护自己（限调用方），漏桶保护下游（限输出）。

场景：接口限流用令牌桶（Guava RateLimiter）；MQ 消费、支付回调用漏桶。

延展：固定窗口有临界问题（0.99s 和 1.0s 各 x 个请求）；滑动窗口更平滑；分布式限流用 Redis+Lua 保证原子性。

MySQL MVCC 可重复读实现原理

数据库 · 大厂高频

MVCC：多版本并发控制，InnoDB 实现可重复读，读写不阻塞。

三大组件：① 隐藏字段：DB_TRX_ID（最近修改该行的事务 ID）、DB_ROLL_PTR（回滚指针，指向 Undo 链）、DB_ROW_ID（无主键时用）；② Undo Log：版本链，存历史版本；③ Read View：m_ids（活跃事务）、min_trx_id、max_trx_id、creator_trx_id。

可见性：TRX_ID==creator 可见；<min 可见（事务已提交）；>=max 不可见；[min,max) 内：在 m_ids 则不可见（未提交），否则可见。不可见则沿 ROLL_PTR 找上一版本。

RR vs RC：RR 事务内首次读生成 Read View 并复用，保证可重复读；RC 每次 SELECT 生成新 Read View，可读到其他事务已提交的数据。

MySQL 索引为什么用 B+ 树

数据库 · 大厂高频

二叉树/红黑树问题：二叉树易退化成链表；红黑树虽平衡但仍是二叉，百万数据约 26 层，每层一次磁盘 IO，性能差。

B+ 树优势：多路平衡，每个节点存多个 key，树高低（1 亿数据约 3 层）；非叶节点只存索引不存数据，单页可存更多 key；叶子节点双向链表，范围查询高效。

B 树 vs B+ 树：B+ 树数据全在叶子，查询稳定；B 树数据在非叶节点，范围查询需中序遍历。

MySQL 事务 ACID 与隔离级别

数据库 · 大厂高频

ACID：原子性（Undo Log 回滚）、一致性（业务约束）、隔离性（MVCC+锁）、持久性（Redo Log）。

隔离级别：读未提交→读已提交→可重复读（默认）→串行化。级别越高一致性越强，并发越低。

并发问题：脏读（读未提交）、不可重复读（同一事务内多次读结果不同）、幻读（行数变化）。

幻读解决：RR 下 Next-Key 锁（行锁+间隙锁）可防幻读，但需 SELECT FOR UPDATE 等加锁读。

Binlog：主从复制、按时间点恢复。

两阶段提交：InnoDB Prepare → 写 Binlog → InnoDB Commit。崩溃恢复时：有 COMMIT 则提交；只有 PREPARED 则查 Binlog 有 XID 则提交、无则回滚。

哈夫曼压缩

情景题

① 扫描统计字符频率 freq[256]；② 小根堆建哈夫曼树：每次取两最小合并，直到剩一个；③ 根到叶左 0 右 1 得编码。

// 建树
PriorityQueue<Node> pq = new PriorityQueue<>((a,b)->a.freq-b.freq);
for (int i = 0; i < 256; i++)
    if (freq[i] > 0) pq.offer(new Node((char)i, freq[i]));
while (pq.size() > 1) {
    Node a = pq.poll(), b = pq.poll();
    pq.offer(new Node(a.freq + b.freq, a, b));
}
// 编码：DFS 左0右1，到叶子记录 code
void encode(Node root, String path, Map<Character,String> codes) {
    if (root.left == null && root.right == null) {
        codes.put(root.ch, path); return;
    }
    encode(root.left, path + "0", codes);
    encode(root.right, path + "1", codes);
}

public int[] constructArray(int n, int k) {

    for (int i = 0; i < n; i++) {
        res[i] = i * k;  // 等差数列 [0, k, 2k, ...]
    }
    return res;
}

public boolean hasPathSum(TreeNode root, int target) {

}

    path.add(root.val);

        res.add(new ArrayList<>(path));
    dfs(root.left, target - root.val);
    dfs(root.right, target - root.val);
    path.remove(path.size() - 1);  // 回溯
}

删除链表倒数第 N 个节点

算法 · LeetCode 19

双指针：快指针先走 n+1 步，然后快慢一起走。快指针到 null 时，慢指针在待删节点的前驱。用 dummy 虚拟头统一处理删除头节点的情况。

public ListNode removeNthFromEnd(ListNode head, int n) {
    ListNode dummy = new ListNode(0, head);
    ListNode fast = dummy, slow = dummy;
    for (int i = 0; i <= n; i++) fast = fast.next;  // 快指针先走 n+1 步
    while (fast != null) {
        fast = fast.next;
        slow = slow.next;
    }
    slow.next = slow.next.next;
    return dummy.next;
}

LRU 缓存实现

算法 · LeetCode 146

LRU：最近最少使用，淘汰最久未访问的数据。HashMap 实现 O(1) 查找；双向链表维护访问顺序，头为最近、尾为最久。get/put 都要把访问的节点移到链表头。

class LRUCache {
    static class Node { int key, val; Node prev, next; Node(int k, int v) { key = k; val = v; } }
    Map<Integer, Node> map;
    Node head, tail;  // 虚拟头尾
    int cap;
    public LRUCache(int capacity) {
        cap = capacity;
        map = new HashMap<>();
        head = new Node(0, 0); tail = new Node(0, 0);
        head.next = tail; tail.prev = head;
    }
    public int get(int key) {
        if (!map.containsKey(key)) return -1;
        Node node = map.get(key);
        remove(node); addFirst(node);
        return node.val;
    }
    public void put(int key, int value) {
        if (map.containsKey(key)) {
            remove(map.get(key));
        } else if (map.size() == cap) {
            map.remove(tail.prev.key);
            remove(tail.prev);
        }
        Node node = new Node(key, value);
        map.put(key, node); addFirst(node);
    }
    void remove(Node node) {
        node.prev.next = node.next;
        node.next.prev = node.prev;
    }
    void addFirst(Node node) {
        node.next = head.next; node.prev = head;
        head.next.prev = node; head.next = node;
    }
}

反转链表

算法 · LeetCode 206

迭代： prev=null，cur=head；每次 next=cur.next，cur.next=prev，prev=cur，cur=next。递归：先反转 head.next，再把 head 接到反转后链表的尾部。

// 迭代 O(n) 时间 O(1) 空间
public ListNode reverseList(ListNode head) {
    ListNode prev = null;
    while (head != null) {
        ListNode next = head.next;
        head.next = prev;
        prev = head;
        head = next;
    }
    return prev;
}

// 递归
public ListNode reverseList(ListNode head) {
    if (head == null || head.next == null) return head;
    ListNode newHead = reverseList(head.next);
    head.next.next = head;
    head.next = null;
    return newHead;
}

二叉树最近公共祖先

算法 · LeetCode 236

后序遍历。若 root 为 null 或等于 p 或 q，直接返回 root。递归左右得 left、right。若 left 和 right 都非空，说明 p、q 在 root 两侧，root 即为 LCA；否则返回非空的那一侧（p、q 在同侧）。

public TreeNode lowestCommonAncestor(TreeNode root, TreeNode p, TreeNode q) {
    if (root == null || root == p || root == q) return root;
    TreeNode left = lowestCommonAncestor(root.left, p, q);
    TreeNode right = lowestCommonAncestor(root.right, p, q);
    if (left != null && right != null) return root;  // p、q 在两侧
    return left != null ? left : right;  // 在同侧，返回非空
}

最长递增子序列

算法 · LeetCode 300

DP：dp[i]=以 nums[i] 结尾的 LIS 长度，dp[i]=max(dp[j]+1) for j<i 且 nums[j]<nums[i]。O(n²)。

贪心+二分：维护 tail[i]=长度为 i+1 的 LIS 的最小末尾元素。遍历时二分找第一个≥当前值的位置替换，长度即答案。O(nlogn)。

// 贪心+二分 O(nlogn)
public int lengthOfLIS(int[] nums) {
    int[] tail = new int[nums.length];
    int len = 0;
    for (int x : nums) {
        int left = 0, right = len;
        while (left < right) {
            int mid = (left + right) / 2;
            if (tail[mid] < x) left = mid + 1;
            else right = mid;
        }
        tail[left] = x;
        if (left == len) len++;
    }
    return len;
}

CPU 多级缓存与缓存一致性

计算机基础 · 计算机组成

多级缓存架构：CPU 与内存速度差异大，采用三级缓存。L1：容量最小（几十～几百 KB）、速度最快（2～4 周期），分指令/数据缓存，各核独占；L2：中等容量与速度（10～20 周期），各核独占；L3：容量较大（几 MB～几十 MB）、速度较慢（20～60 周期），所有核共享。访问顺序：L1 → L2 → L3 → 主存。

缓存一致性问题：多核下每核有独占 L1/L2，一核修改共享数据后，其他核的缓存副本过期。需保证：① 写传播（修改需传播到其他核）；② 事务串行化（所有核看到的写顺序一致）。

MESI 协议：总线窥探 + 写失效。缓存行四状态：M（Modified，已修改未写回）、E（Exclusive，独占且与内存一致）、S（Shared，多核共享）、I（Invalid，失效）。

伪共享：多个独立变量共享同一缓存行（通常 64B），一核修改导致整行失效，影响其他核。解决：字节填充、@sun.misc.Contended。

哈希表冲突解决

计算机基础 · 数据结构

冲突原因：不同 key 经 hash 函数映射到同一桶。

链地址法（拉链法）：每个桶存链表，冲突时追加到链表尾部。实现简单，删除方便；链表过长时退化为 O(n)。优化：链表过长转红黑树（HashMap 8 转树、6 退链表）。

开放寻址：冲突时在表中找下一个空位。线性探测：h+1, h+2...；二次探测：h+1², h+2²...；双重哈希：用第二个 hash 函数。优点：无指针、缓存友好；缺点：删除复杂（需标记 tombstone）、负载因子高时性能下降。

再哈希：冲突时换一个 hash 函数重新计算。

延展：负载因子 = 元素数/桶数，一般 0.75 左右扩容；HashMap 容量为 2 的幂，取模用 hash & (n-1) 替代 %。

红黑树与 AVL 区别

计算机基础 · 数据结构

AVL：严格平衡，任意节点左右子树高度差 ≤ 1。查询 O(log n) 最优；插入/删除可能需多次旋转，维护成本高。

红黑树：近似平衡，满足：① 节点红或黑；② 根和叶子（NIL）为黑；③ 红节点的子必为黑；④ 任意节点到其子孙叶子的路径上黑节点数相同。树高最多 2log(n+1)，查询略慢于 AVL，但插入/删除旋转次数少（最多 3 次）。

选型：读多写少用 AVL（如数据库索引）；写多或需频繁插入删除用红黑树（如 Java TreeMap、Linux 调度器 CFS）。

延展：B+ 树多路平衡，适合磁盘；跳表实现简单，Redis ZSet 底层用跳表+字典。

排序算法总结

计算机基础 · 算法

复杂度与稳定性：

  • 冒泡/选择/插入：O(n²)，冒泡和插入稳定，选择不稳定
  • 快排：平均 O(nlogn)，最坏 O(n²)，不稳定；原地，空间 O(logn)
  • 归并：O(nlogn) 稳定，空间 O(n)；适合外部排序、链表排序
  • 堆排序：O(nlogn) 不稳定，原地

快排 vs 归并：快排平均更快、原地；归并稳定、最坏也 O(nlogn)。工程常用快排，需随机 pivot 或三数取中防退化。

稳定性：相等元素排序后相对顺序不变。多关键字排序时，先按次关键字排，再按主关键字排，需稳定排序。

大端与小端

计算机基础 · 计算机组成

定义：多字节数据在内存中的存放顺序。大端（Big-Endian）：高字节存低地址，如 0x12345678 存为 12 34 56 78；小端（Little-Endian）：低字节存低地址，存为 78 56 34 12。

常见场景：网络字节序为大端（TCP/IP）；x86 为小端；ARM 可配置。网络传输需统一，发送前 htonl/ntohl 转换。

判断方法：用 union 或指针取低地址字节，若为高字节则为大端。

浮点数表示

计算机基础 · 计算机组成

IEEE 754：float 32 位：1 符号位 + 8 指数位 + 23 尾数位；double 64 位：1 + 11 + 52。公式：(-1)^s × 1.M × 2^(E-127)（规格化）。

精度问题：浮点无法精确表示部分十进制（如 0.1），比较用误差范围；累加误差会累积。金融用 BigDecimal。

特殊值：指数全 0 全 1 表示非规格化数、无穷、NaN。

编译与解释

计算机基础 · 编译原理

编译型：源码一次性翻译成机器码（或中间码），再执行。如 C、C++、Go。执行快，但需针对平台编译。

解释型：逐行或逐块解释执行，不生成独立可执行文件。如 Python、JavaScript（传统）。开发灵活，执行较慢。

混合型：先编译成字节码，再由虚拟机解释/JIT 编译。如 Java、C#。JIT 将热点代码编译为机器码，兼顾启动速度和运行性能。

编译流程：词法分析 → 语法分析 → 语义分析 → 中间代码生成 → 优化 → 目标代码生成。
