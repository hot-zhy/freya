TCP 握手是三次，挥手是四次为什么？

计算机网络

三次握手：

  • 防止历史连接：若用两次，旧 SYN 可能迟到，服务端误开连接
  • 同步序列号：双方确认对方的初始序列号
  • 避免资源浪费：两次握手时服务端无法确认客户端是否收到 SYN+ACK

四次挥手：

  • TCP 全双工，双方需各自关闭
  • 被动方收到 FIN 后可能还有数据要发，先 ACK，发完再发 FIN
  • FIN 和 ACK 不能合并

TCP 三次握手和四次挥手的区别，为什么少一次？

计算机网络

握手时服务端 SYN+ACK 可合并；挥手时被动方可能还有数据要发，FIN 不能与 ACK 同时发，所以多一次。

同步阻塞、同步非阻塞、异步阻塞、异步非阻塞的区别

操作系统 / 并发

同步/异步：调用方是否等待结果返回。

阻塞/非阻塞：等待时线程是否挂起。

  • 同步阻塞 (BIO)：调用后一直等，线程阻塞
  • 同步非阻塞 (NIO 轮询)：立即返回，轮询检查是否就绪
  • 异步阻塞：少见
  • 异步非阻塞 (AIO / NIO 多路复用)：注册回调，就绪后通知

乐观锁和悲观锁的区别

并发

悲观锁：先加锁再操作。如 synchronized、SELECT FOR UPDATE、行锁。

乐观锁：无锁，通过版本号/CAS 检测冲突后重试。

读多写少、冲突少时用乐观锁；写多、冲突多时用悲观锁。

缓存雪崩、穿透、击穿的区别与解决方案

Redis / 缓存 · 大厂高频

三者区别：

  • 缓存穿透：查询的数据在缓存和 DB 都不存在，请求绕过缓存直打 DB（非法/恶意查询）
  • 缓存击穿：热点 key 过期，大量并发集中打到 DB（单个热点失效）
  • 缓存雪崩：大量 key 同时失效或 Redis 宕机，请求集中打 DB（大面积失效）

解决方案：

  • 穿透：布隆过滤器、缓存空对象（短 TTL）、参数校验
  • 击穿：互斥锁、热点永不过期、逻辑过期+异步更新、预热
  • 雪崩：过期时间加随机值、多级缓存、限流熔断、Redis 高可用

Redis 热 Key 问题如何发现和解决？

Redis / 缓存 · 大厂高频

什么是热 Key：某 Key 访问量远高于其他，如总 QPS 10000 时某 Key 达 7000。

危害：缓存击穿、单节点网卡打满、DB 雪崩、集群不均衡。

发现：经验预测、redis-cli --hotkeys、MONITOR+redis-faina、客户端/代理层统计。

解决：

  • 本地缓存（Guava/Caffeine）分散请求
  • 热 Key 拆分：key_0001～key_0015 分散到不同节点
  • 热 Key 备份、读写分离、从节点扩容
  • 热 Key 永不过期 + MQ 异步更新

Redis RDB 与 AOF 持久化区别

Redis / 缓存

RDB：二进制快照，某一时刻完整数据。触发：save m n、BGSAVE。恢复快、文件小，但可能丢失分钟级数据。

AOF：追加写命令，重启时重放。刷盘策略：always / everysec（默认）/ no。数据安全高，恢复慢、文件易膨胀。

对比：RDB 恢复快、AOF 数据安全；生产常用混合模式（Redis 4.0+）平衡两者。

限流算法：漏桶 vs 令牌桶

Redis / 缓存 · 大厂高频

漏桶：请求进桶排队，以固定速率流出。强制匀速，无法处理突发。

令牌桶：以恒定速率生成令牌，请求取令牌通过。允许突发（桶有容量）。

区别：令牌桶允许突发、漏桶强制平滑。令牌桶保护自己（限调用方），漏桶保护下游（限输出）。

场景：接口限流用令牌桶；消息队列消费、支付系统用漏桶。

a++ 和 ++a 谁性能高？

Java 某大厂二面

i++ 需保存旧值再 +1；++i 直接 +1。现代 JVM 下字节码层面几乎无差异，JIT 会优化。单条语句中差异可忽略。

在内存有限的机器上尽量使用数组还是链表？

Java / 数据结构 某大厂二面

数组：连续内存、缓存友好、无额外指针开销、O(1) 寻址。

链表：节点分散、每节点 8/16 字节指针、缓存不友好、内存碎片。

内存紧张时数组更优。

Spring Bean 生命周期

Java / Spring · 大厂高频

主要阶段：

  1. 实例化（构造方法创建对象）
  2. 属性注入（依赖注入）
  3. Aware 回调（BeanNameAware、BeanFactoryAware 等）
  4. BeanPostProcessor.postProcessBeforeInitialization
  5. 初始化：@PostConstruct → InitializingBean.afterPropertiesSet() → init-method
  6. BeanPostProcessor.postProcessAfterInitialization
  7. 销毁：@PreDestroy → DisposableBean.destroy() → destroy-method

实例化是分配内存，初始化是执行业务逻辑。BeanPostProcessor 是扩展点。

JVM 垃圾回收机制

Java / Spring · 大厂高频

垃圾识别：可达性分析（从 GC Roots 出发，不可达即垃圾）。GC Roots：栈引用、静态变量、常量、JNI 引用。

回收算法：标记-清除（有碎片）、复制（新生代）、标记-整理（老年代）。

回收器：Serial/ParNew、Parallel、CMS（已废弃）、G1（JDK9+ 默认）、ZGC（超低停顿）。

分代：Minor GC（新生代）、Major/Full GC（老年代/整堆）。

MySQL MVCC 可重复读实现原理

数据库 · 大厂高频

MVCC：多版本并发控制，InnoDB 实现可重复读的机制。

三大组件：

  • 隐藏字段：DB_TRX_ID、DB_ROLL_PTR、DB_ROW_ID
  • Undo Log：版本链，存历史版本
  • Read View：m_ids、min_trx_id、max_trx_id、creator_trx_id

可见性：TRX_ID == creator 可见；< min 可见；>= max 不可见；在 [min,max) 内看是否在 m_ids 中。

RR vs RC：RR 事务开始时生成一次 Read View；RC 每次 SELECT 生成新 Read View。

消息重复消费与幂等性

消息队列 · 大厂高频

原因：网络抖动、Consumer 宕机未 ACK、Rebalance 等导致重复投递。

幂等方案：

  • 唯一键：DB 唯一索引，重复插入失败
  • Redis：messageId 存 Set，存在则跳过
  • 状态机：业务状态流转，幂等校验
  • 分布式锁：messageId 做锁 key，获取成功才处理

核心：用 messageId 或业务唯一键做去重。
